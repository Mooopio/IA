03

XML — THE FOUNDATION

DEFINITION

What is XML?

Extensible Markup Language — a general purpose meta-language for representation, exchange, and processing of data. It's the basis for many web formats.

🔑

KEY WORD: META-LANGUAGE — XML is not a language with fixed tags, it's a language for DEFINING languages. You make your own tags.

XML is the foundation of:

XHTML XSL / XSLT XML Schema RSS Atom RDF (serialization)

📦 Content Types of Elements

TypeWhat it means
EmptyNo content at all — <br/>
SimpleText values only — <isbn>123</isbn>
ElementContains child elements
MixedSimple + Element content both

🌳 Tree Terminology

  • Root element — top-level, only ONE allowed
  • Parent element — contains other elements
  • Child element — inside a parent
  • Siblings — elements at the same level (sister elements)
  • Attribute — name-value pair on an element
<?xml version="1.0" encoding="UTF-8"?> <!-- XML Declaration --> <bookstock> <!-- ROOT element --> <book id="42"> <!-- Attribute: id="42" --> <title>TCP/IP Illustrated</title> <!-- Simple content --> <author>W.R. Stevens</author> <!-- Siblings with title, isbn --> <isbn>0201633469</isbn> </book> </bookstock> <!-- End of document -->

⚖️ XML Syntax Rules (for Well-Formed documents)

RULES — MUST FOLLOW

✅ Well-Formed XML

  • Non-empty elements: start tag + end tag <X>...</X>
  • Empty elements: self-closing <X/>
  • Exactly ONE root element
  • All attribute values must be quoted
  • Tags must NOT overlap: <X><Y></X></Y> ❌
  • XML is case sensitive: <X></x> ❌
  • Content must comply with declared charset (default UTF-8)
COMMON TRAPS

❌ Well-Formed Violations

  • <book><title></book></title> — overlapping!
  • <Book>...</book> — case mismatch!
  • <isbn value=123> — unquoted attribute!
  • Two root elements in one document
  • Missing end tag on non-empty element
🎯

EXAM TRICK: "Well-formed" = follows XML syntax rules. "Valid" = additionally conforms to a DTD or Schema. A valid document is ALWAYS well-formed, but a well-formed document is NOT necessarily valid.

04

XML NAMESPACES

THE PROBLEM

🚨 Why Namespaces Exist

Different contributors use the same element name to mean different things. Example: <title> could mean the document's title OR a book's title in the same file → NAME CONFLICTS.

/* PROBLEM: program scanning for book titles will also find document title */ <datafile> <title>Book List</title> <!-- document title --> <book> <title>TCP/IP Illustrated</title> <!-- book title — CONFLICT! --> </book> </datafile> /* SOLUTION: Bind each group to a unique namespace using a URI prefix */ <datafile xmlns:a="http://example.com/doc/" xmlns:b="http://example.com/myNs/"> a:<title>Book List</a:title> <b:book> <b:title>TCP/IP Illustrated</b:title> <!-- No conflict! --> </b:book> </datafile>
HOW

xmlns: prefix

xmlns:prefix="URI" imports the namespace and binds it to a prefix.

IMPORTANT

URIs ≠ real URLs

The namespace URI is just a unique identifier. It might point to a non-existing resource — that's OK!

USAGE

Standardized dialects

XHTML, XSD, etc. define their own namespace that you import into your documents.

05

XML VALIDATION

DEFINITION

A valid XML document is related to and conforms to further information describing its structure and data types. Two common validation languages: DTD and XML Schema.

Feature DTD (Document Type Definition) XML Schema (XSD)
Format Custom syntax (not XML) Written in XML itself
Data Types Very limited (mostly PCDATA) Rich: string, integer, date, etc.
Complex Types ❌ No complex types ✅ Sequences, choices, groups
Namespace support Limited Full namespace support
Shortcoming Limited type definition capabilities More verbose but powerful

DTD Example

<!DOCTYPE note [ <!ELEMENT book (title,author,isbn)> /* book has 3 children: title, author, isbn */ <!ELEMENT title (#PCDATA)> <!ELEMENT author (#PCDATA)> /* PCDATA = Parsed Character Data = text */ <!ELEMENT isbn (#PCDATA)> ]>

XML Schema (XSD) Example

<xs:element name="book"> <xs:complexType> <xs:sequence> /* sequence = ordered children */ <xs:element name="title" type="xs:string"/> <xs:element name="author" type="xs:string"/> <xs:element name="isbn" type="xs:string"/> </xs:sequence> </xs:complexType> </xs:element>
💡

MEMORY TRICK — DTD vs XSD: DTD = old, simple, Dumb on types. XSD = XML Schema = written IN XML = modern, richer types. Remember: Schema ≥ DTD in every way. DTD's main shortcoming = limited type definitions.

06

XML PROCESSING: DOM vs SAX

MODEL 1

🌳 DOM — Document Object Model

  • Standardized by W3C
  • Loads the entire document into memory as a tree
  • You can traverse the tree or access elements directly by id
  • Supports random access to any node
  • Can read AND modify the document
var elem = document.getElementById("42") var child = elem.getElementsByTagName("author")[0] var val = child.childNode.nodeValue
MODEL 2

⚡ SAX — Simple API for XML

  • Event-based mechanism — not tree-based
  • Reads document sequentially, top to bottom
  • Generates events (startElement, endElement, characters…)
  • You register event handlers that get called
  • Does NOT load whole document — memory efficient
  • Read-only — no modification possible
Aspect DOM SAX
Memory ⚠️ High — loads full tree ✅ Low — sequential read
Access pattern Random access (by id, by tag) Sequential only
Modification ✅ Can modify document ❌ Read-only
API type Tree traversal API Event / callback model
Best for Complex manipulation, searching Large files, simple extraction
Standardized by W3C Open community (saxproject.org)

SAX Event Sequence Example

Start Document
↓
Start Element: bookstock
↓
Start Element: book (Attributes: id=42)
↓
Start Element: title
↓
Characters: "TCP/IP Illustrated"
↓
End Element: title
↓
... (continues sequentially) ...
↓
End Document
🎯

EXAM TRICK: DOM = like a photo (everything in memory at once). SAX = like a movie (events stream one by one). If exam asks "which is better for large files?" → SAX (memory efficient). If "which allows modification?" → DOM only.

07

XML ALTERNATIVES

ALTERNATIVE 1

📋 JSON — JavaScript Object Notation

Discussed in detail in Chapter 3. Lightweight, less verbose than XML, native to JavaScript.

ALTERNATIVE 2

📝 YAML — YAML Ain't Markup Language

  • Compact, text-based, human-readable
  • Superset of JSON (all JSON is valid YAML)
  • Supports: lists, associative arrays, scalars
  • Enables cross-references between nodes
  • Hierarchies by indentation (no tags!)
lectures: - 'Internet and Web Applications' - 'Distributed Systems' - 'Mobile Communication'
⚠️

WHY ALTERNATIVES? XML has overhead and redundant information (verbose opening + closing tags). JSON/YAML are lighter. YAML = superset of JSON — remember this for the exam!

08

XHTML

DEFINITION

Extensible Hypertext Markup Language

XHTML is an XML reformulation of HTML — it takes HTML's concepts but applies strict XML syntax rules.

✅ Advantages of XHTML

  • High standardization → better browser engine unification
  • Processable by regular XML tools (style, validation)
  • Extensible with other XML languages (e.g., MathML for math expressions)
  • Can be validated by DTD (e.g., XHTML 1.1 DTD from W3C)

🔑 XHTML vs HTML Key Difference

  • XHTML = stricter syntax (must be well-formed)
  • Empty elements MUST self-close: <br/> not <br>
  • All tags lowercase, all attributes quoted
  • Expressive power derived from HTML
<?xml version="1.0" encoding="UTF-8"?> <!DOCTYPE html PUBLIC "-//W3C//DTD XHTML 1.1//EN" "http://www.w3.org/TR/xhtml11/DTD/xhtml11.dtd"> <!-- Reference to DTD --> <html xmlns="http://www.w3.org/1999/xhtml" xml:lang="en"> ↑ Default namespace defined here <body> <br/> <!-- Empty elements MUST be self-closed in XHTML --> <table> <tr><td>TCP/IP Illustrated</td><td>W.R. Stevens</td></tr> </table> </body> </html>

JAVASCRIPT

DEFINITION

⚡ What is JavaScript?

  • Object-based, wide-spread script language
  • Standardized as ECMAScript
  • Used primarily for client-side execution
  • Embedded in or linked from (X)HTML documents
  • Browser identifies code → forwards to JavaScript engine
  • Result can change document content or trigger reactions (e.g. open new window)
SECURITY

🔒 Sandbox Principle

Script running in the JS engine can only access objects inside the browser by default.

Cannot access:

  • ❌ File system
  • ❌ Network resources directly
  • ❌ Operating system

Can access:

  • ✅ DOM of the current page
  • ✅ Browser APIs (window, document…)
KEY COMBO

JavaScript + DOM = Dynamic Web Pages

JavaScript is often used with the DOM API to dynamically modify the structure and content of an XHTML document — this is the core of Web 2.0 interactivity (Ajax, etc.)

<html> <head> <script language="JavaScript" type="text/javascript"> function hello() { alert("Hello!"); } </script> </head> <body onLoad="hello()"> <!-- onLoad = called when page loads --> This is a simple JavaScript example. </body> </html>

CSS — CASCADING STYLE SHEETS

DEFINITION

🎨 What is CSS?

Client-side language for defining the layout of markup language elements. Simplifies uniform design and its maintenance of web pages.

How to include CSS

  • Directly embedded in HTML file (<style> tag)
  • Linked externally: <link rel="stylesheet" href="main.css">
💡

KEY BENEFIT: External CSS = change ONE file → update ALL linked pages simultaneously!

CSS Selector Types

SelectorTargetsSyntax
ElementAll <p> tagsp { }
Classclass="top".top { }
IDid="fat"#fat { }
/* main.css — selectors and their properties */ input { /* Element selector — targets all <input> */ font-size: 110%; color: red; background: #ffff00; } .top { /* Class selector — targets class="top" */ padding-top: 20px; } p { /* Element selector */ font-size: 16pt; font-family: "Garamond", serif; font-weight: bold; } #fat { /* ID selector — targets id="fat" */ font-weight: bold; }
🧠

MEMORY TRICK for Selectors: p = plain element. .class = dot for class. #id = hash for ID. In order of specificity: ID (#) > Class (.) > Element. "ID wins everything!"

EXAM QUIZ — TEST YOURSELF

Q01 What are the THREE fundamental mechanisms of the original World Wide Web (Web 1.0)?
A) TCP, IP, DNS
B) HTTP, HTML, URI
C) XML, JSON, YAML
D) Ajax, WebSocket, Comet
✅ HTTP = Application protocol. HTML = Document description format. URI = Document addressing mechanism. These three are the pillars of the original WWW.
Q02 A document that follows XML syntax rules (one root, quoted attributes, no overlapping tags) is called…?
A) Valid
B) Well-formed
C) Structured
D) Validated
✅ Well-formed = correct syntax. Valid = well-formed PLUS conforms to DTD/Schema. Remember: valid ⊃ well-formed.
Q03 Which XML processing model loads the ENTIRE document into memory as a tree structure?
A) SAX
B) DOM
C) YAML
D) XSD
✅ DOM (Document Object Model) interprets the document in its logical tree structure, requiring the entire tree to be loaded into memory. SAX reads sequentially and doesn't load everything.
Q04 What is the main shortcoming of DTD compared to XML Schema?
A) DTD cannot reference external files
B) DTD has limited type definition capabilities
C) DTD is written in XML making it verbose
D) DTD cannot validate element names
✅ The main shortcoming of DTD is limited type definition capabilities — it mostly uses PCDATA (plain text). XSD supports rich types like string, integer, date, etc.
Q05 XML namespaces solve name conflicts using URIs. What's important to know about these URIs?
A) They must point to valid, existing web pages
B) They must use HTTPS protocol
C) They are unique identifiers and may point to non-existing resources
D) They must be registered with W3C
✅ Namespace URIs are just unique identifiers — they DON'T need to point to real resources. They just need to be unique strings that avoid naming conflicts.
Q06 JavaScript's sandbox principle means that scripts running in the browser can access…?
A) The file system and network resources freely
B) Only objects inside the browser, by default
C) Any OS-level resource
D) The user's cookies and passwords directly
✅ Sandbox principle = JS can only access objects inside the browser by default. No direct file system or arbitrary network access. This is a core security concept.
Q07 What is the correct CSS selector for targeting an element with id="header"?
A) .header { }
B) #header { }
C) header { }
D) @header { }
✅ #id selector uses the hash (#) symbol. .class uses dot (.). element selector has no prefix. Specificity: #id > .class > element.
Q08 Web 3.0 / Semantic Web focuses on…?
A) Better social networking features
B) Faster HTTP protocols
C) Semantically interconnected DATA (Linked Data), not documents
D) Improved client-side JavaScript engines
✅ Web 3.0 = Semantic Web = focus on DATA not documents = Linked Data. Uses RDF, RDFS, OWL. It's a CONCEPTUAL evolution (not just technological like Web 2.0).
Q09 YAML is described as a "superset of JSON." What does this mean?
A) JSON can parse YAML directly
B) All valid JSON is also valid YAML
C) YAML is faster than JSON
D) YAML requires JSON to work
✅ Superset means YAML includes everything JSON can do, plus more. So every JSON document is a valid YAML document, but not vice versa.
Q10 SAX (Simple API for XML) is best described as what type of processing model?
A) Tree-based with random access
B) Schema validation model
C) Event-based, sequential processing
D) In-memory DOM modification model
✅ SAX = event-based. The parser reads sequentially, firing events (startElement, characters, endElement, etc.) that your handlers catch. Memory efficient, no random access, read-only.
Q11 What does the Content-Type header do in an HTTP response?
A) Encrypts the content
B) Tells the client which application to use to interpret the content
C) Compresses the response body
D) Sets the document language
✅ Content-Type is meta-information that tells the client the type of content. The client then looks up its "type → application" database and delegates content to the right handler.
Q12 In XHTML, how must empty elements (like line break) be written?
A) <br>
B) <br/>
C) <BR>
D) <break></break>
✅ XHTML follows strict XML rules: empty elements MUST be self-closing with />. In regular HTML you can write <br>, but XHTML requires <br/>.
