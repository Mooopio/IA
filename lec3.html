What is Interaction? — إيه هو الـ Interaction؟

Definition Communication between User↔Application AND Client↔Server — 2 steps
🧠 الـ Interaction ده بيبقى على مستويين: الأول بين الـ User والـ Application، والتاني بين الـ Client والـ Server. يعني لما بتكتب في Google وبتدوس Search، في خطوتين بتحصلوا في نفس الوقت — إنت بتتفاعل مع الـ UI، والـ browser بيتكلم مع الـ Server.
💡
الـ Communication بتاعة الـ Client-Server هي اللي المادة دي كلها عنها — إيه أسرع وأذكى طريقة تبعت وتستقبل Data بيها؟
🔴
Synchronous Model
الموديل الكلاسيكي — "دوس واستنى وشيل الصفحة". المستخدم محجوز لحد ما السيرفر يرد.
🔵
Non-blocking (AJAX)
بعت الطلب وفضلت شغال — مش بيستنى. بس كل ريسبونس محتاج ريكويست قبله.
🟣
Asynchronous (Comet)
السيرفر بيبعت Data وقت ما يحب من غير ما الـ Client يطلب. Real-time فعلي.
🟢
WebSockets
خط مفتوح بين Client وServer — بيكلموا بعض في أي وقت من غير Overhead.
1

Synchronous Communication Model 🔴

🔴 "Click, Wait, and Refresh" — الموديل القديم

Core Definition Synchronous: user actions are BLOCKED until an HTTP Response is received. High coupling between Request and Response — every data transfer requires a preceding request.
🐌 Synchronous يعني إيه؟ يعني لما تعمل أي حاجة — زي ما تدوس زرار — كل حاجة بتتجمد وتستنى لحد ما السيرفر يرد. زي ما تكون بتكلم حد على التليفون، قلتله كلام، وقعدت صامت تستنى رده — مش قادر تعمل أي حاجة تانية.
📋 مثال ملموس — فورم Registration:
١- كتبت Username → دوست Submit
٢- الـ Browser بعت HTTP Request للـ Server
٣- الصفحة كلها اتجمدت تستنى
٤- السيرفر رد بـ HTTP Response فيه صفحة HTML كاملة جديدة
٥- الـ Browser شال الصفحة كلها وحطها من أول وجديد — حتى الحاجات اللي ما اتغيرتش!

Synchronous Flow — كل request بيجيب صفحة كاملة جديدة

⚠️ مشاكل الموديل ده (Shortcomings)

🐢
Slow Performance: كل حاجة بسيطة محتاجة تبعت HTTP Request وتستنى Full Page — بطئ جداً. تخيل كل ما تكتب حرف في Search Box الصفحة تتشال!
💔
Loss of Operation Context: لما الصفحة تتشال وترجع، بتفقد كل اللي كنت فيه — مثلاً لو كنت Scroll تحت، ترجع للأول. Scroll position، form data اللي ما اتبعتش — كله بيتمسح.
📡
Excessive Server Load & Bandwidth: بيبعت الصفحة كاملة كل مرة حتى لو بس حاجة صغيرة اتغيرت — ده بيحمل السيرفر أكتر من اللازم ويستهلك Bandwidth زيادة.
📝
النتيجة النهائية: "Slow, unreliable, low productivity, low interactivity and inefficient web applications" — ده الملخص الرسمي في الـ Slide. ⚠️ امتحان

Non-Blocking Model — AJAX 🔵

📜 تاريخ AJAX — المهم تعرفه

📅 التاريخ ده مهم جداً في الامتحانات، خليك فاكره:
1
1999 — Microsoft أطلقت "Outlook Web Access" كأول تطبيق بيستخدم مبدأ AJAX. في الوقت ده كان اسمه XMLHttpRequest. بس بس IE بس اللي كان بيدعمه.
2
~2002 — Mozilla و Netscape طبّقوا الـ XMLHttpRequest في Mozilla 1.0 و Netscape 7 — بس النجاح كان محدود.
3
2005 — الانفجار الحقيقي: Jesse James Garrett كتب مقاله الشهير وأطلق اسم "AJAX". وGoogle Suggest و Google Maps اشتهروا باستخدام AJAX → Gained Popularity.
🎯
حاجة مهمة جداً: IE5 و IE6 كانوا بيستخدموا ActiveXObject مش XMLHttpRequest الـ native. IE7 بس اللي بدأ يدعم الـ native XMLHttpRequest. ده سؤال كتير بيجي في الامتحانات! ⚠️ امتحان

🔵 AJAX — تعريف Jesse James Garrett

Official Definition "AJAX isn't a technology. It's really several technologies, each flourishing in its own right, coming together in powerful new ways."
🧩 AJAX مش تكنولوجيا واحدة! ده مجموعة تكنولوجيات متجمعة مع بعض:

AJAX = Bundle of Technologies

🔵 كيف AJAX بيشتغل — الـ AJAX Engine

Key Concept AJAX introduces an intermediary called the "AJAX Engine" — responsible for rendering UI AND communicating with server on user's behalf. User actions generate JavaScript calls to the engine, NOT HTTP Requests directly.
⚙️ فكرة AJAX بيحط وسيط "AJAX Engine" بين الـ User وبين الـ Server. بدل ما كل كليكة تبعت HTTP Request وتستنى، الـ User دلوقتي بيكلم الـ Engine بـ JavaScript Call، والـ Engine هو اللي بيقرر امتى يكلم السيرفر.

AJAX Architecture — User actions → Engine → Server (maybe)

💡
الفرق الجوهري: الـ AJAX Engine ممكن يعمل حاجات من غير ما يكلم السيرفر خالص! زي: Data Validation (الـ email صح؟) أو Navigation أو Editing data in memory. بس لو محتاج data من السيرفر، بيبعت Request Asynchronously — يعني مش بيوقف الـ User.
🎯
نقطة امتحان مهمة جداً! الـ AJAX يقولوا عليه "Non-blocking" مش "Asynchronous" الحقيقية لأن لكل Server-side update لازم يحصل Full Request/Response round-trip. يعني في High Dependence بين الـ Request والـ Response. ⚠️ امتحان

XMLHttpRequest Object ⚡ — قلب AJAX

Definition XMLHttpRequest provides a client-side programming API for transfer of data via HTTP behind the scenes — without causing a complete page refresh.
❤️ الـ XMLHttpRequest ده هو قلب AJAX. ده Object بيديك API تبعت بيه HTTP Requests في الـ Background من غير ما الصفحة تتشال. ببساطة: ده هو الحاجة اللي بتخليك تكلم السيرفر "سري".

🏗️ إزاي تعمل XMLHttpRequest Object

📄 الطريقة الصح — Compatible مع كل المتصفحات
function getXMLHttpRequest() {
  // Modern browsers (IE7+, Firefox, Chrome, Safari)
  if (window.XMLHttpRequest) {
    return new XMLHttpRequest();
  }
  // IE6 and below — uses ActiveX!
  else if (window.ActiveXObject) {
    try {
      return new ActiveXObject("Msxml2.XMLHTTP"); // IE5, IE6
    } catch(e) {
      try {
        return new ActiveXObject("Microsoft.XMLHTTP"); // Older IE
      } catch(e) { return null; }
    }
  }
  return false;
}
🔍 بنبص الأول هل window.XMLHttpRequest موجود (المتصفحات الحديثة)، لو لأ بنجرب ActiveXObject (IE القديم). الـ try-catch عشان لو النسخة الأولى ما نجحتش نجرب التانية.

📋 Methods — الأوامر

Method 1 open("method", "URL", [async, userName, password]) — Pre-sets up a connection. async defaults to FALSE!
🔧 open() بتحضر الـ Connection — بتقوله:
- الـ Method هايبعت بيها (GET أو POST)
- الـ URL اللي هايبعت عليها
- async بـ default = false! يعني لو ما كتبتش true، هيبقى synchronous
⚠️
نقطة خطيرة جداً! الـ async parameter الـ Default بتاعه هو false. يعني لو كتبت req.open("GET", "url") من غير الـ true، هيبقى Synchronous وهيحجز الـ User! لازم تكتب req.open("GET", "url", true) عشان يبقى Async. ⚠️ امتحان
Method 2 send("data") — Activates the open() settings and sends the request. GET → send(null). POST → send("key=val&key2=val2")
🚀 send() بيبعت الـ Request فعلاً. لو GET → تبعت null. لو POST → تبعت الـ Data زي: "page=1§ion=4"
Method 3 abort() — Cancels the current request
🛑 abort() ببساطة بيلغي الـ Request اللي ماشي.

📊 Properties — الخصائص المهمة جداً

Key Property onreadystatechange — Event handler pointing to the callback function. Fires every time readyState changes.
📡 onreadystatechange ده Event Handler. كل ما الـ readyState يتغير، الـ Function اللي حاطها هنا بتتنفذ. هو قلب الـ Async processing!
Critical Property — readyState Returns current HTTP Request processing status. Five values: 0=Uninitialized, 1=Loading, 2=Loaded, 3=Interactive, 4=Complete

readyState Values — الحالات الخمسة

Value Name ما اللي بيحصل
0 Uninitialized الـ Object اتعمل بس الـ open() ما اتكلتش عليه لسه
1 Loading الـ open() اتكلت بس الـ send() لسه ما اتبعتش
2 Loaded الـ Request اتبعت
3 Interactive جزء من الـ Response وصل — ناقص
4 Complete كل الـ Data وصلت والـ Connection اتغلق ✅
🎯
دايماً بتشيك على: readyState == 4 && status == 200 — يعني "الـ Response كامل وناجح". ٤ تعني Complete و٢٠٠ هو HTTP Status Code بمعنى OK. ⚠️ امتحان
Response Properties responseText → response as string | responseXML → response in XML format | status → numeric HTTP code | statusText → text of HTTP code
📥 بعد ما يكتمل الـ Response عندك:
- responseText: الـ Response كـ String عادي
- responseXML: الـ Response كـ XML Document جاهز تتعامل معاه بالـ DOM
- status: الكود الرقمي (200, 404, 500...)
- statusText: النص المقابل ("OK", "Not Found", "Internal Server Error"...)

💻 AJAX Hello World Example — مثال كامل

📁 المثال فيه ٣ ملفات: helloworld.html (الـ UI) + helloworld.js (الـ Logic) + helloworld.xml (الـ Data). الفكرة: لما تضغط Button، يجيب الـ XML ويعرضه في الصفحة من غير Refresh.
📄 helloworld.html — الـ UI
<html>
<head>
  <script type="text/javascript" src="helloworld.js"></script>
</head>
<body>
  <!-- زرار بيستدعي الـ loadData() -->
  <input type="button" onclick="javascript:loadData();"
         value="Get XML data"/>
  
  <!-- المكان اللي هيتحدث من غير Refresh -->
  <div id="present"></div>
</body>
</html>
📄 helloworld.xml — البيانات
<?xml version="1.0" encoding="UTF-8"?>
<helloworld>
  <message><lang>Chinese</lang><text>你好世界!</text></message>
  <message><lang>English</lang><text>Hello world!</text></message>
  <message><lang>French</lang><text>Salut le monde!</text></message>
</helloworld>
📄 helloworld.js — الـ Logic الكاملة
function loadData() {
  // ١- جيب Object
  var req = getXMLHttpRequest();
  
  // ٢- حضّر الـ Request (GET, URL, async=true)
  req.open("GET", "helloworld.xml", true);
  
  // ٣- حدد الـ Callback — هتتنفذ كل ما readyState تتغير
  req.onreadystatechange = function() {
    // ٤- اشتغل بس لما الـ Response كامل وناجح
    if(req.readyState == 4 && req.status == 200) {
      
      // ٥- حط Table جوه الـ div
      document.getElementById("present").innerHTML = 
        "<table><tr><td id=\"content\"></td></tr></table>";
      
      var output = document.getElementById("content");
      var xml = req.responseXML;  // XML Document
      
      // ٦- Loop على الـ childNodes
      for(var i=0; i<xml.documentElement.childNodes.length; i++) {
        
        // ⚠️ Firefox: فراغات بتبقى TEXT_NODE — skip them!
        if(xml.documentElement.childNodes[i].nodeType == 3) continue;
        
        // اجيب الـ lang text
        var langTag = xml.documentElement
          .childNodes[i]
          .getElementsByTagName("lang")[0]
          .firstChild.nodeValue;
        
        output.appendChild(document.createTextNode(langTag));
        output.appendChild(document.createTextNode(" -- "));
        
        // اجيب الـ text value
        var textTag = xml.documentElement
          .childNodes[i]
          .getElementsByTagName("text")[0]
          .firstChild.nodeValue;
        
        output.appendChild(document.createTextNode(textTag));
        output.appendChild(document.createElement("br"));
      }
    }
  };
  
  // ٧- ابعت الـ Request (GET → null)
  req.send(null);
}
🦊
Firefox Gotcha! الـ nodeType == 3 check ده مهم جداً! Firefox بيفسر الـ spaces والـ line breaks بين الـ XML tags كـ TEXT_NODE. لو ما عملتش الـ check ده، الـ loop هيعطيك Error عشان هيحاول يعمل getElementsByTagName على Text Node. ⚠️ امتحان

JSON — البديل الأخف 📦

Why JSON? X in AJAX = XML. But XML creates unnecessary overhead. JSON is: lightweight, compact, language-independent, faster to parse than XML.
⚖️ الـ X في AJAX يعني XML — بس XML فيها overhead زيادة (Tags طويلة، Parsing بطيء). الـ JSON بقى هو البديل الأشهر دلوقتي لأنه:
• Compact — أقل bytes، أقل bandwidth
• Lightweight — Parsing أسرع من XML
• Language-independent — بيتعامل معاه أي لغة
• Familiar — بيستخدم syntax قريبة من C-family

🏗️ JSON Structure — التركيب

JSON Building Blocks

📄 JSON Example — نفس الـ Hello World بس بـ JSON
{
  "helloworld": [
    { "lang": "English", "text": "Hello World!" },
    { "lang": "French",  "text": "Salut le Monde!" },
    { "lang": "German",  "text": "Hallo Welt!" }
  ]
}
// هنا "helloworld" هو Object key، وقيمته Array من Objects
// كل Object جواه فيه "lang" و "text"
⚖️
XML vs JSON — المقارنة السريعة: نفس الـ Data، الـ JSON أقصر وأسرع في الـ Parsing. الـ JavaScript بتعامل معاه بشكل Native بـ JSON.parse().

Asynchronous Model — Comet 🟣

Core Concept Comet enables web servers to send data to the client with only an initial HTTP Request — without the client needing to request data again. Completely asynchronous → low-latency data transfer.
📡 المشكلة في AJAX إنه لكل بيانات جديدة محتاج الـ Client يبعت Request جديد. Comet حل ده — الـ Client بيبعت Request واحد بس، والسيرفر بيفضل يبعت Data في أي وقت من غير ما يستنى حد يطلبه. = Real-time push!
📅
التاريخ مهم: في March 2006، Alex Russell (project leader of Dojo Toolkit — Ajax framework) صاغ مصطلح "Comet". Comet مش اختصار لحاجة! زي AJAX، ده buzzword. ⚠️ امتحان
🎯
Scope of Application — إمتى تستخدمه؟ التطبيقات اللي محتاج فيها Real-time data زي: Chat applications، Stock exchange (البورصة)، eBay next-gen (يعرفك أحدث عرض من غير ما تضغط Refresh).

⚙️ إزاي Comet بيشتغل — Long-Polling

Comet Long-Polling — خط مفتوح مع السيرفر

🔑 الـ Comet بيعمل:
• Long-living HTTP Connection — بيفتح Connection واحد وبيفضله مفتوح
• Server Push — السيرفر بيبعت Data وقت ما يجيله Event
• لو الـ Client هيبعت Data للسيرفر، محتاج Connection تاني منفصل
• ده بيتسمى برضو "HTTP Streaming"
⚠️
مشاكل Long-Polling:
١- Overhead من HTTP Headers
٢- Inefficiency — الـ Connection مفتوح حتى لو ما فيش أي events
→ الحل: WebSockets! ⚠️ امتحان

WebSockets 🟢 — الحل الأمثل

❓ ليه WebSockets؟ مشاكل HTTP Streaming

HTTP Streaming Shortcomings 1. Overhead due to HTTP headers and unnecessary HTTP requests in long-polling. 2. Communication channel is UNIDIRECTIONAL — separate HTTP requests needed for bidirectional exchange.
🔧 مشكلتين رئيسيتين في الـ HTTP-based solutions:
١- Overhead: كل HTTP Request فيه Header كبيرة — لو بتبعت messages صغيرة كتير، الـ Header أكبر من الـ Data نفسها!
٢- Unidirectional: الـ HTTP Connection بيتفتح في اتجاه واحد. عشان تبعت وتستقبل في نفس الوقت محتاج Connections منفصلة.
WebSockets Solution Standard for TCP-based BIDIRECTIONAL communication channels. Associated with HTML5. Widely supported in modern browsers and web servers. Defined by IETF in RFC 6455 (Protocol) and W3C (JavaScript API).
🚀 الـ WebSockets بيعمل:
• TCP-based — بيستخدم TCP مباشرة مش HTTP فوقيه
• Bidirectional (Full-Duplex) — Client وServer يكلموا بعض في نفس الوقت على نفس الـ Connection
• HTML5 Standard — مش تجربة، ده Standard رسمي
• RFC 6455 بيعرف الـ Protocol، W3C بيعرف الـ JavaScript API

🤝 WebSocket Handshake — كيف يتفتح الـ Connection

Process WebSocket connection established using existing TCP connection via HTTP protocol. Three steps: Client sends HTTP Upgrade Request → Server responds with HTTP 101 Switching Protocols → WebSocket messages exchanged.
1
HTTP Upgrade Request: الـ Client بيبعت HTTP Request عادي بس فيه Header خاص Upgrade: websocket يقول "عايز أترقى لـ WebSocket"
2
HTTP 101 Switching Protocols: السيرفر لو موافق بيرد بـ Status Code 101 — يعني "تمام، بنتحول"
3
WebSocket Connection Established: دلوقتي عندك خط مفتوح TCP، تقدر تبعت وتستقبل Binary أو UTF-8 messages في أي اتجاه في أي وقت!

🔐 Handshake Details — التفاصيل الفنية

📤 Client Request
GET /chat HTTP/1.1
Host: server.example.com
Upgrade: websocket          ← اللي بيقول "عايز WebSocket"
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==  ← Base64 Random String
Origin: http://example.com
Sec-WebSocket-Protocol: chat, superchat  ← App-specific protocols
Sec-WebSocket-Version: 13
📥 Server Response
HTTP/1.1 101 Switching Protocols     ← 101 = Switch accepted!
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=  ← Derived from Key
Sec-WebSocket-Protocol: chat   ← Protocol accepted
🔑
الـ Key/Accept Logic — مهم جداً:
الـ Client بيبعت Sec-WebSocket-Key وهو Base64-encoded random string.
السيرفر بياخد الـ Key ده + بيضيفله string ثابت هو: "258EAFA5-E914-47DA-95CA-C5AB0DC85B11"
بيعمل SHA-1 hash للـ concatenation ده
بيعمل Base64-encode للـ hash
النتيجة دي هي الـ Sec-WebSocket-Accept
→ ده بيثبت إن السيرفر فاهم WebSocket وموافق على الـ Connection. ⚠️ امتحان

💻 WebSocket JavaScript API

📄 WebSocket API — الـ Events والـ Methods
// إنشاء WebSocket connection
var ws = new WebSocket("ws://localhost:9998/chat");
// لاحظ: ws:// مش http://

// Event: Connection opened
ws.onopen = function() {
  ws.send("Hello Server!");  // بعت message
  alert("Connected & message sent!");
};

// Event: Message received FROM server
ws.onmessage = function(evt) {
  var received = evt.data;  // الـ message اللي جت
  alert("Received: " + received);
};

// Event: Connection closed
ws.onclose = function() {
  alert("Connection closed");
};

// Event: Error occurred
ws.onerror = function(error) {
  console.error("WebSocket error: " + error);
};

// إغلاق الـ connection
ws.close();
📝
الـ 4 Events: onopen / onclose / onerror / onmessage
الـ 2 Methods: send(string) / close()
الـ URL بتبدأ بـ ws:// (أو wss:// للـ secure) مش http:// ⚠️ امتحان

الفنيات والحاجات اللي بتلغبط 🎯

⚠️ أخطر الـ Tricky Points — اللي بيلغبط في الامتحان

⚡ 1. async Default = false !

الـ open() method الـ Parameter التالت async الـ Default بتاعه false — مش true!
لو كتبت req.open("GET", "url") من غير true → هيبقى Synchronous وهيحجز الـ User!
دايماً لازم تكتب: req.open("GET", "url", true)

⚡ 2. AJAX مش Asynchronous حقيقية!

رغم إن الاسم فيه "Asynchronous"، الـ AJAX يتوصف في الـ Lecture بـ "Non-blocking" مش "Asynchronous" كاملة.
السبب: فيه High Dependence بين Request وResponse — كل response من السيرفر محتاج Request كامل قبله.
بينما Comet وWebSockets دول اللي فيهم Minimum Dependence — يعني الـ True Asynchronous.

⚡ 3. IE Browser History — مهم جداً

IE5/IE6: بيستخدموا new ActiveXObject("Msxml2.XMLHTTP") أو "Microsoft.XMLHTTP"
IE7+: بيدعم native new XMLHttpRequest()
الترتيب في الكود: الأول نشيك على window.XMLHttpRequest، لو مش موجود نشيك على window.ActiveXObject

⚡ 4. Firefox nodeType==3 Bug

لما بتقرأ XML في JavaScript، Firefox بيعتبر الـ whitespace والـ newlines بين الـ Tags كـ TEXT_NODE (type 3).
لو عملت loop على الـ childNodes من غير check، هتاخد Error لما تحاول تعمل getElementsByTagName على Text Node.
الحل: if(node.nodeType == 3) continue;

⚡ 5. readyState==4 && status==200

مش بس readyState==4 كفاية! لازم كمان status==200 (OK).
ممكن تاخد readyState==4 مع status==404 (Not Found) أو 500 (Server Error).
لو شيكت بس على readyState، هتعمل معالجة على response فاشل!

⚡ 6. Comet للـ Upload محتاج Connection تاني

الـ Comet Long-polling Connection مخصوص فقط للـ Data Push (Server → Client).
لو الـ Client عايز يبعت Data للسيرفر، محتاج يفتح Connection HTTP تاني منفصل.
ده من مشاكل HTTP Streaming اللي الـ WebSockets حلتها.

⚡ 7. WebSocket URL — ws:// مش http://

الـ WebSocket URL بيبدأ بـ ws:// مش http://.
للـ secure version: wss:// (زي https للـ HTTP).
مثال: new WebSocket("ws://localhost:9998/chat")

⚡ 8. Sec-WebSocket-Key Algorithm

📚 EXAM SUMMARY — اللي يخليك تقفل أي امتحان

🔴 1. Synchronous Model — "Click, Wait, Refresh"

  • User actions are BLOCKED until HTTP Response received
  • Every data transfer needs a preceding HTTP Request (HIGH coupling)
  • Server sends a COMPLETE HTML page every time
  • Problems: Slow, loss of context, excessive server load, redundant refreshes
  • Result: "slow, unreliable, low productivity, low interactivity and inefficient"

🔵 2. AJAX — Non-Blocking Model

  • Introduced by Microsoft (1999, Outlook Web Access) as "XMLHttpRequest" — IE only
  • Mozilla/Netscape added native support ~2002 in their browsers
  • Named "AJAX" by Jesse James Garrett in 2005 — popularized by Google Suggest/Maps
  • AJAX = NOT a single technology — bundle of: XHTML+CSS, DOM, XML+XSLT, XMLHttpRequest, JavaScript
  • Introduces "AJAX Engine" as intermediary — handles UI + server communication
  • User actions → JS calls to Engine (NOT directly HTTP Requests)
  • Engine can process locally (validation, navigation) without calling server
  • Still HIGH dependence: each server update needs full request/response round-trip
  • So AJAX = "Non-blocking", NOT truly asynchronous!
  • IE5/IE6: ActiveXObject("Msxml2.XMLHTTP") | IE7+: native XMLHttpRequest

⚡ 2b. XMLHttpRequest — The Heart of AJAX

  • Methods: open(method, URL, [async=false!]) | send(data) | abort()
  • open() async parameter DEFAULT = false — must set true for async!
  • GET requests: send(null) | POST requests: send("key=val&key2=val2")
  • Properties: onreadystatechange | readyState (0-4) | responseText | responseXML | status | statusText
  • readyState values: 0=Uninitialized, 1=Loading (open called), 2=Loaded (sent), 3=Interactive (partial), 4=Complete
  • Always check: readyState==4 && status==200
  • Firefox issue: whitespace between XML tags = TEXT_NODE (type 3) → must skip with continue

📦 2c. JSON vs XML

  • X in AJAX = XML — but XML creates unnecessary overhead
  • JSON = JavaScript Object Notation: lightweight, compact, faster parsing
  • JSON structures: Object {} (unordered name/value pairs) | Array [] (ordered collection)
  • JSON values: string, number, boolean (true/false), object, array — can be nested
  • Language-independent, familiar to C-family programmers

🟣 3. Comet — Asynchronous Model (Long-Polling)

  • Named by Alex Russell (Dojo Toolkit leader) in March 2006
  • Comet is NOT an acronym — just a buzzword (like AJAX)
  • Server sends data WITHOUT client needing to request it → truly asynchronous
  • Works via "long-living HTTP connections" that stay open
  • Server-side events trigger data push to client
  • Also called "server push" or "HTTP streaming"
  • For uploading: requires a second separate HTTP connection
  • Use cases: chat, stock exchange, real-time notifications
  • Shortcomings: HTTP header overhead, inefficiency of long-polling

🟢 4. WebSockets — The Complete Solution

  • Defined by IETF in RFC 6455 (Protocol) | W3C defines JavaScript API
  • Associated with HTML5
  • TCP-based bidirectional (full-duplex) communication
  • Established using HTTP Upgrade mechanism (reuses existing TCP connection)
  • Steps: HTTP Upgrade Request → Server: 101 Switching Protocols → WebSocket messages
  • Handshake: Client sends Sec-WebSocket-Key (Base64 random) → Server concatenates with magic GUID → SHA-1 → Base64 = Sec-WebSocket-Accept
  • Magic GUID: "258EAFA5-E914-47DA-95CA-C5AB0DC85B11"
  • API: new WebSocket("ws://...") | Events: onopen, onclose, onerror, onmessage | Methods: send(), close()
  • URL scheme: ws:// (or wss:// for secure) — NOT http://
  • Messages can be Binary or UTF-8
  • Fixes both Comet problems: no HTTP overhead, true bidirectional!
