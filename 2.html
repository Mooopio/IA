01 — REST Model

Representational State Transfer

REST is an architectural style for distributed hypermedia systems, especially the Web. Think of it like ordering food at a restaurant — the menu is the interface, each dish is a resource, and every order message contains everything the waiter needs.

📌 Definition

REST = Representational State Transfer. A model based on client/server architecture with stateless communication, where requests target resources (not procedures) via a generic interface.

Key REST Principles

  • Stateless: Every message must contain all necessary context. The server stores no session state between requests.
  • Resources not procedures: Unlike RPC (Remote Procedure Call) which calls functions, REST targets documents/resources.
  • Unique identifiers: Every resource has a unique URI.
  • Client/Server separation: Clear division of concerns.
🔑 Exam Alert — REST vs RPC

REST requests are directed to resources (documents) using standard HTTP methods. RPC requests are directed to procedures (functions). This is a classic MCQ trap.

REST Interaction Flow

1
GET document

Client requests a resource using its unique URI. Server returns its representation.

2
Client manipulates

Client processes the received representation locally.

3
PUT document

Client sends the modified representation back to the server at the same URI.

02 — Web App Architecture

Three-Tier Architecture

A web application has three logical layers. Think of it like a restaurant: the waiter (presentation), the chef (logic), and the pantry (data).

LayerLocationResponsibility
PresentationClient (browser)Render UI, handle user interaction
LogicServer-sideProcess requests, business rules, authentication
DataServer-sidePersistent storage, databases
⚠️ Evolution Trend

Modern web apps shift logic increasingly to the client-side (JavaScript frameworks). Classical apps kept almost everything server-side. This is a classic exam trajectory question.

03 — Web Server Architecture

Hierarchical Processing Pipeline

A web server makes resources available via HTTP. For performance under high load, it uses a pool of pre-created processing entities — like a fast-food chain staffing multiple cashiers.

📌 Definition

A web server is software that makes resources available through an HTTP-accessible interface. It uses a hierarchical, parallel structure for low latency and high throughput.

Request Handling Steps

1
Startup: Pool creation

Server creates a pool of processing entities and opens a port for incoming connections.

2
Browser connects via TCP

Browser initiates a TCP connection to the Connector Entity.

3
Client Descriptor created

The Connector Entity creates a client descriptor (the communication endpoint) and passes it to the Delegation Process.

4
Delegation assigns work

Delegation Process selects an available processing entity from the pool and forwards the client descriptor.

5
HTTP communication

The selected processing entity handles full HTTP communication with the browser.

Dynamic Content Generation

Static content: server just reads from filesystem. Dynamic content uses:

  • CGI scripts — Simple external programs
  • Java Servlet Container — Java-based processing
  • Application Server — Full services: transactions, security, directory

An (X)HTML engine can merge dynamic content with templates to produce final HTML pages.

05 — HTTP Protocol

Hypertext Transfer Protocol

HTTP is the communication protocol of the Web. It runs on top of TCP using a request-response pattern. Think of it like sending a formal letter: every letter has a header (to, from, subject) and a body (content).

📌 Structure of Every HTTP Message

Header: specifies the operation, resource URI, and parameters (key-value pairs)
Body: optional content (HTML, form data, file, etc.)
Separated by a blank line (CRLF)

URI Addressing

  • * — No resource associated (e.g. OPTIONS *)
  • http://www.serv.com/index.html — Absolute URI (includes server address)
  • /index.html — Absolute Path (relative to server, requires Host header)

HTTP 1.0 vs HTTP 1.1 — Critical Differences

FeatureHTTP 1.0HTTP 1.1
ConnectionClose after each requestPersistent (keep-alive) by default
PipeliningNot availableAvailable (but not universally supported)
Host headerOptionalRequired with absolute paths
MethodsGET, HEAD, POST+PUT, DELETE, OPTIONS, TRACE, CONNECT
Efficiency (packets)497 packets total223 (persistent), 83 (pipelining)
🔑 Pipelining

HTTP/1.1 pipelining: send multiple requests without waiting for responses. Responses are returned in order. Reduces from 497 packets (HTTP/1.0) to just 83 packets with pipelining.

Servers can adapt content to different devices by reading the User-Agent header. Similarly, Accept-Language enables language-specific responses.

06 — Session Handling

Maintaining State in Stateless HTTP

HTTP is stateless — each request is independent. But web apps need to remember users (shopping carts, login state). The solution: Sessions + Cookies.

📌 How It Works

Server creates a session data structure and assigns a session ID (SID). The SID is sent to the browser as a cookie. Browser includes the cookie in all future requests. Server uses the SID to retrieve the right session data.

Cookies

Small pieces of data stored client-side in the browser's data persistence layer. Each cookie has a name, value, and source domain. Same-origin policy applies.

07 — HTTP/2

Next-Generation Protocol

HTTP/1.1 wasn't designed for performance. HTTP/2 fixes this — think of HTTP/1.1 as a single-lane road vs HTTP/2 as a multi-lane highway.

FeatureHTTP/1.1HTTP/2
MultiplexingOne request at a time per connectionMultiple parallel channels per TCP connection
Header CompressionNoneYes (HPACK)
FramingText messagesBinary frames (HEADER + DATA frames)
PrioritizationNot availableFrames can be prioritized
SemanticsHTTP methods/codesSame — no change to HTTP semantics
🔑 Critical Detail

HTTP/2 does NOT change HTTP methods, status codes, or semantics. It only optimizes transport. Approved as IETF standard in February 2015.

Two messages are split into 4 interleaved frames: two HEADER frames and two DATA frames, transmitted over one TCP connection via separate channels.

08 — HTML5

The Modern Web Platform

HTML5 = Enhanced HTML + CSS3 + JavaScript APIs. It bridges client-side and server-side capabilities.

📌 HTML5 = 3 Parts

Enhanced HTML — New structural elements and attributes
CSS3 — Improved styling, animations, responsive design
JavaScript APIs — Web Storage, Web Workers, Geolocation, IndexedDB

Structural Elements

ElementPurpose
<header>Introductory or navigational aids
<nav>Navigation section
<section>Generic grouping element
<article>Independent content (blog post, news)
<aside>Tangentially related content (sidebar)
<footer>Footer info (author, copyright)
⚠️ Trap

These elements are for grouping/structure, NOT for positioning. Positioning is done with CSS.

Audio & Video

HTML5 introduces <audio> and <video> elements with built-in JS APIs. Key features: multiple source formats, text track support (WebVTT subtitles), JavaScript play/pause control.

Web Workers

📌 Definition

A Web Worker is JavaScript that runs in the background, independently of the UI thread. Communication is event-based using postMessage(). Used for computationally expensive tasks that would otherwise freeze the UI.

// Main thread
var worker = new Worker("counter_code.js");
worker.onmessage = function(event) { /* handle result */ };

// Worker file
postMessage(value); // sends back to main thread

Client-Side Storage (HTML5)

Storage TypeDescriptionAdvantageDisadvantage
Web StorageKey/value pairsSimple API, wide supportNo queries, no schemas, no transactions
IndexedDBObject stores with indexes/cursorsPowerful, simplifies DB programmingNot in all browsers (historically)
Web SQL DatabaseSQL in browserFull SQL (joins, selects)Spec STOPPED by W3C
🔑 Web Storage Details

sessionStorage: Lives as long as the browser tab/window is open. NOT shared across tabs.
localStorage: No time limit. Shared across tabs/windows from the same origin. Fires a storage event when modified.

All HTML5 storage uses the same-origin policy: scripts can only access data stored by scripts from the same protocol + hostname + port.

IndexedDB

For larger structured data with high-performance key/index/cursor-based searches. Operations are asynchronous — results come via DOM events (onsuccess, onerror handlers).

var request = window.indexedDB.open("BookDB");
request.onsuccess = function(event) {
  var database = event.result;
};

⚠️ Common Traps & Mistakes
Trap 1

REST vs RPC: REST targets resources (documents). RPC targets procedures (functions). Many students confuse these.

Trap 2

HTML5 structural elements are for structure/grouping only — NOT for visual positioning. CSS handles layout.

Trap 3

HTTP/2 changes NOTHING about HTTP semantics, methods, or status codes. It only changes how messages are transported.

Trap 4

sessionStorage ≠ localStorage: sessionStorage is tab-scoped (gone when tab closes). localStorage is permanent and cross-tab within same origin.

Trap 5

Web SQL Database was ABANDONED by W3C. Don't recommend it. IndexedDB is the current standard.

Trap 6

Pipelining order: Responses must come back in the same order as requests were sent. This is a known limitation.

Trap 7

HEAD method: Identical to GET but the response body is empty. Use case: get file metadata (size, type) without downloading the file.
