02
The Web: 1.0 → 2.0 → 3.0
تطور الويب من البداية للسيمانتيك ويب
World Wide Web (Web 1.0) الويب الأساسي
The World Wide Web هو نظام من Hypertext Documents (مستندات فيها روابط لمستندات تانية) بيشوفها المتصفح.

Hypertext = نص تشعبي – يعني مش نص عادي، لكن فيه روابط داخلية لمستندات تانية. زي إنك بتقرأ مقال على ويكيبيديا وفيه لينكات تودي لمقالات تانية – ده Hypertext.

الويب بيشتغل فوق بروتوكولات الإنترنت – يعني الإنترنت هو الأساس (زي الطريق)، والويب هو تطبيق يشتغل عليه (زي العربية على الطريق).
3 ركائز أساسية للويب:

1️⃣ HTTP (Hypertext Transfer Protocol): البروتوكول اللي المتصفح بيتكلم بيه مع السيرفر. زي اللغة المشتركة بينهم.

2️⃣ HTML (Hypertext Markup Language): اللغة اللي بيتكتب بيها محتوى الصفحة. السيرفر بيبعت HTML، والمتصفح بيرسمها على الشاشة.

3️⃣ URI (Uniform Resource Identifiers): "عنوان" كل مورد على الإنترنت. زي https://example.com/page – ده URI بيحدد مكان الصفحة.
Web 2.0 الويب التفاعلي
Web 2.0 مش بروتوكول جديد أو تكنولوجيا منفصلة – هو buzzword (مصطلح تسويقي) بيوصف الأشكال الحديثة للتطبيقات والخدمات اللي بتعمل في المتصفح.

مثال: الفرق بين موقع أخبار قديم (بتقرأ بس) وبين Facebook (بتقرأ، تكتب، تتفاعل، تشير) – ده الفرق بين Web 1.0 و Web 2.0.

خصائص Web 2.0:
✅ Social component عالي – ناس بتتكلم مع ناس
✅ User interfaces جميلة – مش بس HTML عادي
✅ Client-side logic بتزيد – كتير من الكود بيشتغل في المتصفح مش السيرفر
⚠️ نقطة امتحان مهمة
المحاضرة بتقول صراحة إن Web 2.0 هو technological evolution (تطور تكنولوجي) بينما Web 3.0 هو conceptual evolution (تطور مفاهيمي). الفرق ده بيجي في الامتحان!
Web 3.0 – The Semantic Web ويب الفهم والمعنى
Web 3.0 أو Semantic Web هي رؤية (vision) لمستقبل الويب حيث البيانات على الإنترنت بتكون مترابطة ومفهومة للآلات، مش بس للبشر.

الفرق التاريخي:
📄 Web 1.0: بنشتغل مع مستندات (Documents) – صفحة HTML بتقرأها
👥 Web 2.0: برضو مستندات بس مع تفاعل اجتماعي
🔗 Web 3.0: بنشتغل مع بيانات (Data) – Linked Data – البيانات نفسها ليها معنى ومرتبطة ببعض

مثال عملي: لو دكتور كتب إنه من "القاهرة"، Web 3.0 يفهم إن "القاهرة" دي عاصمة مصر وموقعها الجغرافي 30°N، بدل ما يبقى مجرد نص "القاهرة" مش بيفهمه أوتوماتيكي.

Formats مهمة: RDF, RDFS, OWL, Microformats

03
XML – Extensible Markup Language
لغة وصف البيانات القابلة للتوسع
What is XML? إيه هو XML؟
XML (Extensible Markup Language) هي meta-language – يعني "لغة عن اللغات" – بتستخدمها عشان توصف بياناتك وتبادلها ومعالجتها بشكل منظم.

ليه بنقول "Extensible"؟ لأنك بتعرّف tags (وسوم) بنفسك بحسب احتياجاتك. مش زي HTML اللي فيها tags محددة (<h1>، <p>)، في XML تقدر تعمل <book>، <student>، أي حاجة.

XML هي أساس كتير من formats على الويب:
XML Terminology & Structure مصطلحات XML والهيكل الشجري
مستندات XML بتتنظم في هيكل شجري (tree structure) – تخيل شجرة عائلة، كل عنصر (Element) ممكن يكون فيه عناصر جوه جوه.

أنواع محتوى العنصر:
🔸 Empty content: العنصر فاضي خالص، زي <br/>
🔸 Simple content: نص عادي جوه العنصر، زي <title>TCP/IP</title>
🔸 Element content: عناصر جوه عناصر، زي <book><title>...</title></book>
🔸 Mixed content: نص وعناصر مع بعض
XML Example – من المحاضرة بالضبط 
<!-- XML Declaration - بتحدد الإصدار والـ encoding -->
<?xml version="1.0" encoding="UTF-8"?>

<!-- Root Element - عنصر جذر واحد بس -->
<bookstock>

  <!-- Child Element مع Attribute -->
  <book id="42">
    <title>TCP/IP Illustrated</title>   <!-- Simple Content -->
    <author>W.R. Stevens</author>
    <isbn>0201633469</isbn>
  </book>

  <book id="43">
    <title>Mobile Web Services</title>
    <author>F. Hirsch</author>
    <author>J. Kemp</author>   <!-- عنصرين بنفس الاسم = OK -->
    <isbn>0470015969</isbn>
  </book>

</bookstock>

<!-- End of document -->
مصطلحات لازم تحفظها:

🌳 Root Element: عنصر الجذر – <bookstock> – واحد بس في الملف
👨‍👧 Parent/Child: <bookstock> هو Parent لـ <book>
👫 Siblings: عناصر على نفس المستوى – <title>، <author>، <isbn> هم siblings
🏷️ Attribute: خاصية جوه start tag – id="42"
📝 Simple Content: النص الجوه العنصر – "TCP/IP Illustrated"
XML Syntax Rules – Well-Formed قواعد بناء XML الصحيح
لو مستند XML متبع القواعد دي يتسمى "Well-formed" (مبني صح). القواعد دي لازم تعرفها عشان الامتحان بيجيب أسئلة "إيه المشكلة في الكود ده؟":
✅ CORRECT
<book>content</book>
<br/>  ← empty element
<X><Y></Y></X>  ← nested OK
id="42"  ← quoted attribute
<Book>=<Book>  ← same case
❌ WRONG
← No end tag:
<book>content
← Two root elements:
<a></a><b></b>
← Overlapping:
<X><Y></X></Y>  ❌
← Unquoted:
id=42  ❌
← Case mismatch:
<X></x>  ❌
الفرق بين Well-Formed و Valid
Well-Formed: بيلتزم بقواعد XML الأساسية (كل tag مفتوح يتقفل، root واحد، attributes بين quotes)
Valid: Well-Formed + بيتماشى مع DTD أو XML Schema اللي بيحدد هيكل معين

⚠️ Well-Formed أولاً، Valid تانياً. مستند ممكن يكون well-formed بس مش valid.
XML Namespaces فضاءات الأسماء – حل مشكلة التعارض
المشكلة: تخيل إنك عندك ملف XML فيه بيانات من مصدرين مختلفين. الاتنين عندهم عنصر اسمه <title>، بس الأول يقصد بيه عنوان الكتاب والتاني يقصد اسم الملف. لو اتحطوا في نفس الملف، الكمبيوتر هيتلخبط.

الحل: Namespaces – بنضيف "لاحقة" (prefix) لكل عنصر بتوضح مصدره وتمنع التعارض.
XML Namespaces – قبل وبعد 
=== قبل: تعارض في الأسماء ===
<datafile>
  <title>Book list</title>         ← عنوان الملف؟
  <book>
    <title>TCP/IP Illustrated</title> ← عنوان الكتاب؟
  </book>
</datafile>

=== بعد: namespace يحل المشكلة ===
<datafile 
  xmlns:a="http://www.example.com/doc/"
  xmlns:b="http://www.example.com/myNs/">
  
  <a:title>Book list</a:title>         ← ينتمي للـ namespace (a)
  <b:book>
    <b:title>TCP/IP Illustrated</b:title> ← ينتمي للـ namespace (b)
  </b:book>
</datafile>
ملاحظات مهمة:
🔑 الـ namespace identifier هو URL زي http://www.example.com/doc/ – بس مش لازم يكون موقع حقيقي! مجرد unique identifier (معرّف فريد)
🔑 بنربط الـ namespace بـ prefix قصير (a, b) لتسهيل الكتابة
🔑 كل frameworks مشهورة زي XHTML بتعرّف namespace خاص بيها يتضاف في كل مستند
XML Validity: DTD vs XML Schema التحقق من صحة XML
Valid XML = ملف بيتبع قواعد هيكل معين محدد مسبقاً. في طريقتين لتعريف الهيكل ده:
1. DTD
Document Type Definition – قديمة وبسيطة. بتعرّف قائمة بالعناصر المسموحة وعلاقاتها.

❌ Main shortcoming: قدرات محدودة جداً في تعريف أنواع البيانات 
<!DOCTYPE note [
<!ELEMENT book (title,author,isbn)>
<!ELEMENT title (#PCDATA)>
<!ELEMENT author (#PCDATA)>
<!ELEMENT isbn (#PCDATA)>
]>
// PCDATA = character data (نص)
2. XML Schema
XML Schema – حديثة وأقوى. هي نفسها XML وبتعرّف أنواع بيانات معقدة.

✅ Simple types (string, integer...)
✅ Complex types (sequences of elements) 
<xs:element name="book">
  <xs:complexType>
    <xs:sequence>
      <xs:element 
        name="title" 
        type="xs:string"/>
      <xs:element 
        name="author" 
        type="xs:string"/>
    </xs:sequence>
  </xs:complexType>
</xs:element>
نقطة فارقة في الامتحان
DTD بتتكتب بصيغة مختلفة عن XML (مش XML). أما XML Schema فهي نفسها XML dialect – ملف XML بيصف ملف XML تاني! ده بيخليها أقوى وأكثر مرونة. لو الدكتور سألك ليه XML Schema أحسن من DTD، الإجابة: أنواع بيانات أكثر + هي نفسها XML.
04
XHTML – Extensible HyperText Markup Language
HTML بتاعت XML
XHTML – HTML meets XML الجمع بين HTML وXML
XHTML هي إعادة صياغة HTML كـ XML. يعني نفس قدرات HTML (عناوين، روابط، صور، جداول) بس بقواعد XML الصارمة.

فوايد XHTML:
✅ قواعد أكثر صرامة = المتصفحات بتتعامل معاها بشكل أكثر اتساق (stricter syntax → further unification of browser engines)
✅ تقدر تعالجها بأدوات XML العادية (زي validators، XSLT للـ styling)
✅ قابلة للتوسع – تقدر تضيف لغات XML تانية زي MathML لو حبيت تكتب معادلات رياضية
✅ يمكن التحقق منها بـ DTD
XHTML Example – من المحاضرة 
<!-- XML Declaration -->
<?xml version="1.0" encoding="UTF-8"?>

<!-- Reference to DTD - بيقول مين بيحكم هيكل الملف ده -->
<!DOCTYPE html PUBLIC "-//W3C//DTD XHTML 1.1//EN"
  "http://www.w3.org/TR/xhtml11/DTD/xhtml11.dtd">

<!-- Default namespace - إجباري في XHTML -->
<html xmlns="http://www.w3.org/1999/xhtml" xml:lang="en">
<head>
  <title>XHTML Example</title>
</head>
<body>
  <h3>Available books</h3>
  <br/>   <!-- Empty element → directly closed! -->
  <table>
    <tr><td>Title</td><td>Author</td></tr>
    <tr><td>TCP/IP Illustrated</td><td>W.R. Stevens</td></tr>
  </table>
</body>
</html>
XHTML vs HTML – فروق دقيقة بتجي في الامتحان
الموضوع HTML عادي XHTML
Empty elements<br><br/> ← لازم تقفلها!
Attributesتقدر تكتب بدون quotesلازم بين quotes دايماً
Caseغير حساسحساس – كله lowercase

DOM & SAX – XML Processing Models
طريقتان لمعالجة XML
DOM – Document Object Model معالجة XML كشجرة في الذاكرة
DOM (Document Object Model) هو API (واجهة برمجية) معياري من W3C بيخليك تدخل على أي جزء من الـ XML وتعدل فيه.

الفكرة: DOM بيحمل الملف كله في الذاكرة (RAM) كشجرة من الأوبجكتات. بعدين تقدر تتنقل في الشجرة دي أو توصل لأي عنصر مباشرة بالـ id.

مثال: لو عندك XML فيه 1000 كتاب وعايز تعدل الكتاب رقم 42 – مع DOM تروح له مباشرة وتغير بياناته.
DOM API – JavaScript Example 
// الخطوة 1: روح للعنصر بالـ id مباشرة
var elem = document.getElementById("42")
// → عدد رحلة لـ bookstock → book id=42

// الخطوة 2: جوه book، جيب أول عنصر "author"
var child = elem.getElementsByTagName("author")[0]
// → getElementsByTagName returns a LIST of elements
// → [0] يجيب الأول في القائمة

// الخطوة 3: الـ text node جوه author
var secondChild = child.childNode

// الخطوة 4: الـ value (النص فعلاً)
var value = secondChild.nodeValue
// → "W.R. Stevens"
⚠️ عيب DOM: لازم يحمل الملف كله في RAM. لو الملف كبير جداً (مئات الميجا) = مشكلة!
SAX – Simple API for XML معالجة XML بالأحداث – بدون تحميل الملف كله
SAX (Simple API for XML) هو نظام معالجة مختلف تماماً – Event-based (مبني على الأحداث).

الفكرة: بدل ما تحمل الملف كله في الذاكرة، الـ SAX parser بيقرأ الملف سطراً سطراً من الأول للآخر. كل ما يلاقي عنصر أو نص أو غيره، بيولّد "event" (حدث) وبيبعته لـ "handler" (مُعالج) بتكتبه إنت.

Web Interaction & Content Type (MIME)
كيف الكلاينت والسيرفر بيتكلموا
Client-Server Interaction Model نموذج التفاعل على الويب
الويب مبني على Client-Server Architecture. الكلاينت (المتصفح) بيطلب مستندات من السيرفر بـ URI عن طريق HTTP. المستند الواحد ممكن يكون محتوياته من أنواع مختلفة:
Content Type / MIME Type نوع المحتوى – بيقول للمتصفح "إيه ده؟"
Content Type (أو Media Type أو MIME Type) هو metadata (معلومة تانوية) بتيجي مع كل ملف بيبعته السيرفر وبتقول للكلاينت نوع المحتوى ده إيه عشان يعرف يفسره صح.

Format: Content-Type: type/subtype
HTTP Response Header – بيحتوي على Content-Type 
HTTP/1.1 200 OK
Content-Type: application/xml    ← XML عام
Content-Length: 136713
...

== أمثلة أنواع محتوى ==
Content-Type: application/xml    ← XML
Content-Type: text/html           ← HTML عادي
Content-Type: text/plain          ← نص عادي
Content-Type: image/jpeg          ← صورة JPEG
Content-Type: video/mpeg          ← فيديو MPEG
Content-Type: application/json    ← JSON data
إزاي بيشتغل الموضوع؟
1️⃣ الكلاينت يستقبل الرد من السيرفر مع الـ Content-Type في الـ header
2️⃣ المتصفح يبص على النوع – لو هو مسؤول عنه يعرضه مباشرة
3️⃣ لو مش مسؤول، يدور في جدول (database) عنده على تطبيق مناسب يفتح الملف ده
4️⃣ يشغّل التطبيق ده ويعطيه المستند
Client Logic – Pseudo Code من المحاضرة 
if(notResponsible) {
  document = response;
  format   = response.getType();           // جيب نوع المحتوى
  program  = database.lookup(format);     // دور على برنامج مناسب
  application = start(program);           // ابدأ البرنامج
  application.delegateTo(document);      // ادي البرنامج الملف
}
// مثال: لو Content-Type: application/pdf
// المتصفح هيشغل Adobe Acrobat أو PDF viewer

JavaScript
لغة التفاعلية على الجانب العميل
What is JavaScript? إيه هو JavaScript وليه بنستخدمه؟
JavaScript هي لغة سكريبت (script language) قائمة على الأوبجكتات (object-based) ومتشهرة جداً. اسمها القياسي الرسمي هو ECMAScript.

بتتنفذ على جانب الكلاينت (Client-side) – يعني في المتصفح بتاع المستخدم، مش على السيرفر.

بيتضاف للـ HTML بطريقتين:
✅ Embedded: مكتوب مباشرة جوه الـ HTML داخل <script> tag
✅ Linked: ملف .js منفصل بتستدعيه
JavaScript in HTML – المثال من المحاضرة 
<html>
  <head>
    <script language="JavaScript" type="text/javascript">
      function hello() { 
        alert("Hello!");    // بيعمل popup box
      }
    </script>
  </head>
  <body onLoad="hello()">   <!-- لما الصفحة تحمل، نادي على hello() -->
    This is a simple JavaScript example.
  </body>
</html>
// المتصفح بيلاقي script tag → يبعت للـ JavaScript Engine → ينفذ الكود
Sandbox Principle – Security! مبدأ الصندوق الرملي – الأمان
ده مفهوم أمان مهم جداً! المحاضرة بتقول: JavaScript engine بيسمح للكود بس يوصل للـ objects جوه المتصفح – مش أكثر.

الـ JavaScript مش تقدر:
❌ توصل لـ File System (ملفات الكمبيوتر) مباشرة
❌ تتصل بأي Network resource (بروتوكول أمان: Same-Origin Policy)

ليه ده مهم؟ لأنك لو بتفتح موقع مجهول وفيه JavaScript، الكود ده مش يقدر يقرأ ملفاتك أو يتصل بسيرفرات تانية ويسرق بياناتك – الـ sandbox بيحميك.
JavaScript + DOM = Dynamic Pages الجمع بين JS وDOM = صفحات ديناميكية
JavaScript بتستخدم كتير مع DOM – ده بيخلي هيكل مستند XHTML يتغير بشكل ديناميكي (من غير ما الصفحة تعمل reload).

مثال: لما بتضغط على زرار في Gmail وترسالة تتحرك للـ trash من غير ما الصفحة تعمل refresh – ده JS بتعدل في الـ DOM مباشرة.
08
CSS – Cascading Style Sheets
لغة تصميم صفحات الويب
What is CSS? إيه هو CSS وليه بنستخدمه؟
CSS (Cascading Style Sheets) هي لغة client-side لتعريف تصميم وشكل عناصر HTML.

أهم فايدة: Separation of Concerns – فصل المحتوى (HTML) عن التصميم (CSS). بدل ما تكتب اللون والخط في كل element على حدة، بتعرف كل ده في ملف CSS منفصل.

الفايدة العملية: لو بدلت شيء في ملف CSS واحد → كل صفحات الموقع المرتبطة بيه بتتغير تلقائياً!
CSS Selectors & Properties المحددات والخصائص
CSS بيشتغل بنظام Selectors: بتختار عناصر HTML وبتحدد ليها خصائص تصميمية.

أنواع Selectors:
🔵 Element Selector: p { color: red; } ← كل عناصر <p>
🟡 Class Selector: .top { padding: 20px; } ← كل elements بـ class="top"
🟢 ID Selector: #fat { font-weight: bold; } ← عنصر واحد بـ id="fat"
main.css – من المحاضرة 
/* Element selector - كل input */
input {
  font-size: 110%;
  color: red;
  background: #ffff00;  ← أصفر
}

/* Class selector - class="top" */
.top { 
  padding-top: 20px; 
}

/* Element selector - كل p */
p {
  font-size: 16pt;
  font-family: "Garamond", serif;
  font-weight: bold;
}

/* ID selector - id="fat" */
#fat { 
  font-weight: bold; 
}
index.html – استخدام CSS 
<head>
  <!-- ربط ملف CSS -->
  <link rel="stylesheet" 
        href="main.css">
</head>
<body>
  <div class="top">
    <!-- 20px padding من فوق -->
    <img src="/banner.jpg"/>
  </div>
  <p class="top">
    Welcome...  <!-- Garamond, bold -->
  </p>
  <div id="fat">
    Please enter... <!-- bold -->
  </div>
</body>
لماذا "Cascading"؟
"Cascading" (متتالي/متراكب) لأن لو عندك أكتر من مصدر CSS (ملف خارجي + style داخل HTML + inline style)، فيه ترتيب أولوية بينهم يحدد مين بياخد الأسبقية. الـ inline style دايماً بياخد أعلى أولوية.

Tricky Parts & Common Exam Traps
النقاط الصعبة والفخاخ اللي تتجنبها
⚠️ TRAP 1 – Well-Formed vs Valid
Well-formed = بيتبع قواعد XML الأساسية (tags متقفلة، root واحد...) – مش بتحتاج DTD أو Schema
Valid = well-formed + بيتبع كمان schema أو DTD محدد
→ لو الدكتور قالك ملف "well-formed" مش يعني إنه "valid" بالضرورة!
⚠️ TRAP 2 – Namespace URIs مش لازم تكون موقع حقيقي
المحاضرة قالت صراحة: "These identifiers are web addresses that might point to non existing resources" – يعني الـ namespace URI زي http://www.example.com/doc/ مش لازم يكون موقع شغال، هو بس unique identifier (معرّف فريد).
⚠️ TRAP 3 – Web 2.0 هو buzzword مش تكنولوجيا
المحاضرة قالت صراحة: "Web 2.0 is a buzzword for describing modern forms of applications" – مش بروتوكول جديد أو version منفصلة!
Web 2.0 = technological evolution (تطور تكنولوجي)
Web 3.0 = conceptual evolution (تطور مفاهيمي) – الفرق ده بيجي في الامتحان.
⚠️ TRAP 4 – DOM loads ALL document in memory
DOM بيحمل الشجرة كلها في RAM = مشكلة مع الملفات الكبيرة
SAX بيقرأ تتابعياً بدون تحميل الكل = أحسن للملفات الكبيرة بس يقدر يقرأ بس مش يعدل
→ الاختيار بين DOM وSAX بيعتمد على حجم الملف واحتياجك للتعديل
⚠️ TRAP 5 – JavaScript Engine Sandbox
الـ JavaScript بتاعت الويب مش تقدر توصل للـ File System أو Network resources مباشرة بـ default – الـ sandbox بيحميك من الكود الضار.
لو سألك الدكتور "ليه JavaScript آمن؟" = Sandbox Principle
⚠️ TRAP 6 – XML DTD بصيغة مختلفة عن XML
DTD بتتكتب بصيغة خاصة مختلفة عن XML (<!ELEMENT...>)
بس XML Schema هي نفسها XML! ملف XSD هو ملف XML عادي.
ده من أسباب تفوق XML Schema على DTD.
🧠 سؤال: إيه الفرق بين URI و URL؟ (سؤال بيجي في الامتحانات)
هما نفس الشيء
URI أعم – URL هو نوع من URI
URL أعم من URI
🧠 سؤال: لو عندك XML ملف كبير جداً (جيجابايت) وعايز تستخرج منه بيانات معينة – إيه الأنسب؟

Exam Cheat Sheet – Summary
ملخص الامتحان – أقوى حاجة فيها

📋 COMPLETE EXAM SUMMARY – Lecture 1

World Wide Web
System of hypertext documents viewable by a browser. Built on 3 pillars: HTTP (application protocol), HTML (document format), URI (addressing). Runs on top of Internet protocols.
Hypertext
Documents with internal cross-references (links) to other documents. Not just static text.
Web 2.0
Buzzword describing modern web apps (wikis, blogs). High social component, appealing UIs. Includes Ajax, Comet, WebSockets, HTML5, RSS, Atom, RDF. Is a technological evolution.
Web 3.0 / Semantic
Vision of semantically interconnected data. Conceptual evolution. Focus on data not documents → Linked Data. Key formats: RDF, RDFS, OWL, Microformats.
XML
General-purpose meta-language for representation, exchange, processing of data. Extensible (you define your own tags). Tree structure. Basis for XHTML, RSS, Atom, RDF, XSL, XML Schema.
Well-Formed XML
Follows basic XML syntax: start+end tags, exactly one root, quoted attributes, no overlapping tags, case-sensitive. Well-formed ≠ Valid!
Valid XML
Well-formed + conforms to a DTD or XML Schema defining allowed structure/types.
DTD vs XML Schema
DTD: simple, limited type definitions, non-XML syntax. XML Schema: powerful, many data types, IS itself XML. XML Schema preferred for complex type definitions.
XML Namespaces
Solve name conflicts by qualifying elements with unique identifiers (URIs). URI doesn't need to point to existing resource — just a unique ID. Prefix (a:, b:) is bound to namespace.
XHTML
XML reformulation of HTML. Stricter syntax (empty elements must close: <br/>, all lowercase, quoted attributes, XML declaration required, namespace required). Validated by DTD. Extensible (add MathML, etc).
DOM
W3C API for XML/HTML. Loads ENTIRE document into memory as a tree. Allows random access, traversal, modification. Bad for very large files (memory intensive).
SAX
Event-based XML processing. Reads sequentially, generates events (startElement, characters, endElement...) — no full tree in memory. Good for large files. Read-only (cannot easily modify).
Content Type (MIME)
Format: type/subtype (e.g., application/xml, image/jpeg, text/html). Sent in HTTP response header. Client uses it to find appropriate app to handle the content.
JavaScript
Object-based scripting language, standardized as ECMAScript. Client-side execution in browser's JS engine. Sandbox principle: can only access objects inside browser (not file system, not external network by default). Often used with DOM for dynamic pages.
CSS
Cascading Style Sheets. Client-side language for layout/styling of markup. External CSS file → change one file affects all linked pages. Uses selectors: element (p{}), class (.top{}), ID (#fat{}). "Cascading" = priority ordering of multiple sources.
YAML
YAML Ain't Markup Language. Compact, human-readable data serialization. Superset of JSON. Uses indentation for hierarchy. Supports: lists, associative arrays, scalars, cross-references.
