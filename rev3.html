What is Interaction?

Communication = 2 Layers

  • Layer 1: Between User ↔ Application (what the user sees/does)
  • Layer 2: Between Client ↔ Server (HTTP under the hood)
🧠 Memory
Think: U→A→C→S (User talks to App, App talks to Server)

The 3 Models

  • Synchronous → Classic Web (click, wait, refresh)
  • Non-Blocking → AJAX
  • Asynchronous → Comet + WebSockets (HTML5)
01

The 3 Models — Side by Side

Model Technology User Actions HTTP Req/Res Coupling Data Sending
Synchronous Classic Web 🔴 BLOCKED (wait for response) HIGH — 1 request = 1 response always Synchronous with user actions
Non-Blocking AJAX 🟢 NOT blocked HIGH — still 1 request = 1 response Asynchronous (user not blocked)
Asynchronous Comet / WebSockets 🟢 NOT blocked MINIMUM (server pushes freely) Asynchronous — minimum dependency
🎯 Key Exam Trick
AJAX is NOT truly asynchronous — it still needs a request before every server response (high coupling). Only Comet/WebSockets achieve true async (server can push anytime).
02

Model 1: Synchronous (Classic Web)

"Click → Wait → Refresh" Paradigm

User Clicks
→
HTTP Request
→
Server processes
→
Full HTML response
→
Browser refreshes

Browser sends HTTP Request → Server processes → Server sends back complete (X)HTML page → Browser refreshes entire screen.

🔴 Shortcomings

  • Slow performance (click, wait, refresh every time)
  • Loss of operation context on page refresh
  • Excessive server load
  • High bandwidth consumption (full page every time)
  • Redundant page refreshes

Criteria (Definition)

  • Synchronous = user actions are BLOCKED until HTTP Response arrives
  • High dependence = for every data transfer from server→client, a request MUST be sent first
  • User activity and server-side processing are coupled
Result: slow, unreliable, low productivity, low interactivity, inefficient web applications
03

Model 2: Non-Blocking (AJAX)

AJAX = Asynchronous JavaScript + XML

Named by Jesse James Garrett in 2005. Introduced the "AJAX engine" as intermediary between user and server.

📅 History Timeline

  • 1999: Microsoft → "Outlook Web Access" — first AJAX-like app using XMLHttpRequest
  • 1999: Only IE supported XMLHttpRequest (via ActiveX: XMLHTTP)
  • ~2002: Mozilla 1.0 and Netscape 7 implemented native XMLHttpRequest
  • 2005: Jesse James Garrett coined "AJAX" in article → Google Suggest + Google Maps → AJAX became popular

AJAX Components (J.J. Garrett's Definition)

🧩 AJAX is NOT one technology — it's several:

  • Standard-based presentation → XHTML + CSS
  • Dynamic display/interaction → DOM (Document Object Model)
  • Data interchange → XML + XSLT
  • Async data retrieval → XMLHttpRequest object
  • JavaScript → binds everything together

🎯 Intentions of AJAX

  • Create apps that look/feel like desktop apps (e.g. AJAXWrite)
  • Extend classic web with additional features (e.g. auto-completion in Google Suggest)
  • Eliminate "click, wait, refresh" → update only part of the page

The AJAX Engine — How it Works

User Action
→
JavaScript call to AJAX Engine
→
Engine decides: need server?

If NO server needed:

  • Simple data validation
  • Editing data in memory
  • Some navigation

If server IS needed:

  • Engine sends HTTP Request asynchronously
  • User is NOT blocked
  • Gets data → updates page partially

Real-World Examples

  • Google Suggest: As you type, requests sent to server → drop-down list of terms + number of results
  • Google Docs: Free browser-based Word alternative; saves in MS Word, OpenOffice, RTF, Text, PDF
⚠️ Critical Point
AJAX Criteria: Asynchronous = user not blocked. BUT still HIGH coupling = for each server-side update, a full request/response round trip is needed. → Not truly async!

XMLHttpRequest — Heart of AJAX

What is XMLHttpRequest?

Client-side programming API that transfers data via HTTP behind the scenes — without causing a complete page refresh. Request is initialized and response is handled through this object.

Creating the Object

// Modern browsers (IE7+, Firefox, Chrome, etc.)
var xmlHttpRequest = new XMLHttpRequest();

// IE5 & IE6 (ActiveX)
var xmlHttpRequest = new ActiveXObject("Msxml2.XMLHTTP");

// Older IE versions
var xmlHttpRequest = new ActiveXObject("Microsoft.XMLHTTP");

📌 Methods

  • open("method", "URL", [async, user, pass]) — pre-sets up connection; method = GET or POST; async default = false
  • send("data") — activates the open() settings; GET → data is null; POST → data is key=value pairs (e.g. page=1§ion=4)
  • abort() — cancels current request

📌 Properties

  • onreadystatechange — event handler → points to callback function; fires when readyState changes
  • readyState — current status (0–4)
  • responseText — response as string
  • responseXML — response as XML
  • status — numeric HTTP status code
  • statusText — text of HTTP status
Note
responseText, responseXML, status, statusText are all extracted from the HTTP Response

readyState — 5 Values (0 to 4)

0
Uninitialized
Object created, but open() NOT called yet
1
Loading
open() called but request NOT sent yet
2
Loaded
Request has been sent
3
Interactive
Partial HTTP Response received
4
Complete
All data received, connection closed ✓
🧠 Exam Must Know
Check for: readyState == 4 && status == 200 → means fully done & successful

Complete Working Example — Full Source

// helloworld.js
function loadData() {
  var req = getXMLHttpRequest();
  req.open("GET", "helloworld.xml", true);  // async=true
  
  req.onreadystatechange = function() {
    if(req.readyState == 4 && req.status == 200) {
      // Create table in <div id="present">
      var tab_content = "<table><tr><td id=\"content\"></td></tr></table>";
      document.getElementById("present").innerHTML = tab_content;
      var output = document.getElementById("content");
      var xml = req.responseXML;
      
      for(i=0; i < xml.documentElement.childNodes.length; i++) {
        // Skip text nodes (Firefox treats whitespace as TEXT_NODE)
        if(xml.documentElement.childNodes[i].nodeType == 3) continue;
        var langTag = xml.documentElement.childNodes[i]
          .getElementsByTagName("lang")[0].firstChild.nodeValue;
        output.appendChild(document.createTextNode(langTag));
        output.appendChild(document.createTextNode(" -- "));
        var textTag = xml.documentElement.childNodes[i]
          .getElementsByTagName("text")[0].firstChild.nodeValue;
        output.appendChild(document.createTextNode(textTag));
        output.appendChild(document.createElement("br"));
      }
    }
  };
  req.send(null);  // GET → null
}

// Cross-browser XMLHttpRequest factory
function getXMLHttpRequest() {
  if (window.XMLHttpRequest) { return new XMLHttpRequest(); }
  else if (window.ActiveXObject) {
    try { return new ActiveXObject("Msxml2.XMLHTTP"); }
    catch(e) {
      try { return new ActiveXObject("Microsoft.XMLHTTP"); }
      catch(e) { return null; }
    }
  }
  return false;
}

The 3 Files (Hello World Example)

  • helloworld.html — presentation; has <div id="present"></div> as reserved block; button calls loadData()
  • helloworld.js — client-side processing; contains the XMLHttpRequest logic
  • helloworld.xml — content; parsed XML with <lang> and <text> tags in multiple languages

JSON — Alternative to XML

X in AJAX stands for XML — but XML creates unnecessary overhead. JSON is the better alternative.

What is JSON?

  • JavaScript Object Notation
  • Lightweight data interchange text format
  • Compact encoding, less overhead than XML
  • Language-independent (but uses C-family conventions)
  • Parsing native structures is faster than parsing XML

JSON Structures

  • Object = unordered name/value pairs → {...}, separated by : and ,
  • Array = ordered collection → [...], values separated by ,
  • Values = string ("..."), number, boolean, object, array → can be nested!

JSON Example — Hello World

{
  "helloworld": [
    { "lang": "English", "text": "Hello World!" },
    { "lang": "French",  "text": "Salut le Monde!" },
    { "lang": "German",  "text": "Hallo Welt!" }
  ]
}

Model 3a: Comet (Asynchronous)

What is Comet?

Named in a March 2006 weblog article by Alex Russell (project leader of Dojo Toolkit — an Ajax framework). Comet stands for nothing (like AJAX, it's a buzzword). Uses already existing technologies.

Note
Comet methodology was used in apps BEFORE the phrase "Comet" was coined.

✅ Key Feature

Server can send data to client with only an initial HTTP Request. After that, NO need for client to request data. → Completely asynchronous / low-latency data transfer.

AJAX high coupling problem → SOLVED ✓

🎯 Scope of Application

  • Applications where available data should be transmitted in real-time
  • Chat applications
  • Stock exchange applications
  • Next-gen eBay (always inform latest info without extra clicks)

How Comet Works (Long-Polling / HTTP Streaming)

  • Establishes initially a long-living HTTP connection → used only for data push (server→client)
  • Data push is triggered by server event output
  • For uploading data to the server → a second connection to HTTP server is necessary
  • Method also named: "server push" or "HTTP streaming"

🔴 Shortcomings of Comet (Long-Polling)

  • Overhead due to HTTP headers
  • Unnecessary HTTP requests in long-polling approaches
  • Communication channel is unidirectional → separate HTTP requests needed for bidirectional exchange

→ These drawbacks led to: WebSockets

WebSockets (HTML5)

What are WebSockets?

  • Standard for TCP-based bidirectional communication channels
  • Avoids HTTP overhead and unnecessary requests
  • Part of HTML5
  • Already widely supported in modern browsers and web servers
  • Established using the existing TCP connection via HTTP protocol
  • Data encapsulated in frames → exchanged in full-duplex mode
  • Supports Binary and UTF-8 messages

Two Specifications

  • Protocol Defined by IETF in RFC 6455 → covers: establishing connection (handshake) + data exchange
  • JavaScript API Defined by W3C

WebSocket Connection — 3 Steps

Step 1: HTTP Upgrade Request (browser→server)
→
Step 2: HTTP 101 Switching Protocols (server→browser)
→
Step 3: WebSocket connection ESTABLISHED (bidirectional!)

Handshake Details (RFC 6455)

CLIENT → SERVER
GET /chat HTTP/1.1
Host: server.example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIH...
Origin: http://example.com
Sec-WebSocket-Protocol: chat, superchat
Sec-WebSocket-Version: 13

Key = arbitrarily generated Base64-encoded string

Protocol = requested app-specific protocols on top of WS

⇄
SERVER → CLIENT
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPL...
Sec-WebSocket-Protocol: chat

Accept = SHA-1 hash of (client key + magic string "258EAFA5-E914-47DA-95CA-C5AB0DC85B11") then Base64-encoded

Accept value indicates server willingness to accept connection

WebSocket JavaScript API (W3C)

// Create WebSocket connection
var ws = new WebSocket("ws://localhost:9998/chat");

// Event listeners
ws.onopen = function() {
  ws.send("Example message...");  // send() method
  alert("Message is sent...");
};

ws.onmessage = function(evt) {
  var received_msg = evt.data;
  alert("Message is received...");
};

ws.onclose = function() {
  alert("Connection is closed...");
};

ws.close();  // close() method

4 Event Listeners + 2 Methods

Events

  • onopen — connection established
  • onclose — connection closed
  • onerror — connection/communication error
  • onmessage — message arrives from server

Methods

  • send(string) — send message to server
  • close() — close the connection

Full Comparison & Conclusion

Feature Synchronous AJAX (Non-Blocking) Comet WebSockets
User Blocked? ✅ YES ❌ NO ❌ NO ❌ NO
HTTP Coupling HIGH HIGH MINIMUM MINIMUM
Server Push? ❌ NO ❌ NO ✅ YES ✅ YES
Bidirectional? ❌ NO ❌ NO ⚠️ 2 connections needed ✅ YES (full-duplex)
Protocol HTTP HTTP HTTP long-poll TCP (via HTTP upgrade)
Page Refresh? Full page Partial only None needed None needed
Standard — Multiple technologies Buzzword (Alex Russell, 2006) RFC 6455 (IETF) + W3C API

❌ Synchronous

Slow, unreliable, low productivity, low interactivity, inefficient

Verdict: Outdated paradigm

⚡ AJAX

Smarter, more responsive, more interactive. No "click, wait, refresh". Partial page updates.

BUT: Still high coupling — not truly async

🚀 Comet + WebSockets

Complete async. Comet shortcoming: overhead & inefficiency. WebSockets: TCP-based bi-directional, solves all Comet drawbacks.
