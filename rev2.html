Evolution of Web Apps

FROM SIMPLE TO COMPLEX — THE SHIFT OF LOGIC

EVOLUTION TIMELINE
1
Classical Web Apps (HTTP 1.0): ALL logic on server. Server has Presentation Engine + Presentation Logic + Business Logic + Data. Client only has its own Presentation Engine. Very server-heavy.
2
Intermediate (HTTP 1.1 era): Presentation logic starts moving to client. Server keeps Business Logic + Data. Client now handles rendering locally.
3
Advanced (HTTP 1.1): Business logic starts moving to client too! Server mainly manages Data. Rich client-side applications emerge (think SPAs).
4
Current Web Apps (HTTP 1.1): Client has Presentation Engine + Presentation Logic + Business Logic. Server mainly just stores/serves Data. Maximum client power.
🧠 EXAM TRICK
The evolution direction: Logic migrates FROM server TO client over time. HTTP 1.0 = dumb client, smart server. Modern = smart client, data server. Remember this direction!
REST Model

REPRESENTATIONAL STATE TRANSFER — KNOW EVERY WORD

DEFINITION

What is REST?

A model for distributed hypermedia systems that fits applications in the World Wide Web. It defines HOW client and server talk to each other.

Full name: Representational State Transfer (REST)
KEY PROPERTY

Stateless Communication

Based on client/server architecture with a stateless communication protocol.

Every message contains ALL necessary context information. Neither server NOR client has to store context data!
REST vs RPC — CRITICAL DIFFERENCE
RPC (Remote Procedure Call)

Requests are directed to procedures (functions). You call a specific function on the server. Tightly coupled to implementation.

REST

Requests are directed to resources (documents) using a generic interface with standard semantics. Every resource has a unique identifier (URI).

🧠 MEMORY TRICK
RPC = call a function | REST = access a document/resource via URI. REST is like addressing mail — you address a location, not a person's action.
REST WORKFLOW — 4 STEPS
1
GET document: Client requests a resource from the server using its URI.
2
Server processes: Server handles the request and prepares the resource representation.
3
Manipulate document: Client modifies the resource locally (changes its representation).
4
PUT document: Client sends the modified resource back to the server to update it.
EVERY RESOURCE RULE

In REST, every resource MUST be available through a unique identifier. No identifier = not REST. This is like every webpage having its own URL.

Web Server

ARCHITECTURE + SERVER-SIDE LOGIC

Definition: Software that makes resources available via HTTP
STATIC CONTENT SERVER

Simple Flow

1
Browser sends Request for document
2
Web Server sends Request to File System
3
File System delivers document to Web Server
4
Web Server delivers document to Browser
WHY HIERARCHICAL?

Parallel Processing

Due to the need for low latency and to handle high load, current web servers are structured in a hierarchical way that allows parallel request processing.

Simple sequential processing would be too slow for real-world traffic!
WEB SERVER ARCHITECTURE — 5 COMPONENTS
1
On start-up: Web server creates a pool of processing entities for client handling AND opens an interface for incoming connection requests. Pre-generates them = faster response later.
2
Browser connects via TCP through the interface — handled by the Connector Entity.
3
Connector Entity creates a client descriptor (endpoint of communication path to client) and passes it to the Delegation Process.
4
Delegation Process chooses an available pre-generated processing entity from the pool and forwards the client descriptor to it.
5
Communication with the web browser is finally realized through HTTP.
🧠 MEMORY ORDER
Pool → TCP → Connector → Client Descriptor → Delegation → Processing Entity → HTTP
Remember: "Pool Created First, Then Connect, Describe Client, Delegate, Process, Communicate"

HTTP Deep Dive

HTTP 1.0 vs 1.1 · METHODS · STATUS CODES · MESSAGES · SESSION

HTTP = Hypertext Transfer Protocol (RFC 7230–7235)
HTTP BASICS
Realizes communication on top of TCP by exchange of messages in request-response manner.
Every message divided into: HEADER (sub-headers, key-value params, operations, URI addressing) + BODY (content)
Resources addressed by URI (Uniform Resource Identifier):
• "*" = no resource associated with request
• Absolute URI = e.g. http://www.serv.com/index.html
• Absolute Path = e.g. /index.html
HTTP 1.0 vs HTTP 1.1
HTTP 1.0 — OLD

Each request: Establish TCP → Send Request → Get Response → Close TCP. Then repeat for EVERY resource. Extremely inefficient — new connection each time!

HTTP 1.1 — IMPROVED

Default: persistent connection ("connection: keep-alive"). One TCP connection handles multiple requests. Much more efficient!

HTTP 1.1 PIPELINING

Pipelining = send further requests WITHOUT waiting for responses of previous requests.

Responses transmitted in the ORDER in which associated requests were sent.
⚠️ NOT supported by all common browsers!

HTTP Messages
REQUEST MESSAGE STRUCTURE
1
Request Line: Method + resource URI + HTTP version. Ex: GET /news/index.html HTTP/1.1
2
General Header: Connection type (close/keep-alive), date of request generation.
3
Request Header: Info about request + client (e.g., accepted content encoding).
4
Entity Header: Meta info about body content (e.g., content length). If body available.
5
CRLF: Blank line separating headers from body.
6
Message Body: Request content (e.g., parameters passed to server). Optional.
RESPONSE MESSAGE STRUCTURE
1
Status Line: HTTP version + numeric + textual status code. Ex: HTTP/1.1 200 OK
2
General Header: Connection type, date of response generation.
3
Response Header: Additional response info (e.g., web server product data).
4
Entity Header: Meta info about body (content length, encoding).
5
CRLF: Blank line.
6
Message Body: The actual content (e.g., HTML data of requested page).
Order of header content elements is IRRELEVANT — both request and response headers.
HTTP Methods
Methods specify the resource by passing its absolute or relative URI. If no resource needed, asterisk (*) is passed.
MethodDescriptionHTTP 1.0?
GETRequests delivery of the specified resource✓ YES
HEADSame as GET but message body is EMPTY — client receives only meta info from header. Use: determine file type/length without downloading.✓ YES
POSTSubmits message body content to the specified resource (e.g., form data)✓ YES
PUT*Instructs server to make passed content available under specified URL✗ NEW
DELETE*Deletes the specified resource✗ NEW
OPTIONS*Requests communication options from server (e.g., supported methods)✗ NEW
TRACE*Echoes back the sent request — used for diagnostic purposes✗ NEW
CONNECT*Reserved name for use by proxy servers✗ NEW
🧠 EXAM TRICK — METHODS
HTTP 1.0 had only: GET, HEAD, POST
HTTP 1.1 ADDED: PUT, DELETE, OPTIONS, TRACE, CONNECT (marked with * in slides)
Memory: "1.0 = GHP | 1.1 adds PD-OTC" (Put, Delete, Options, Trace, Connect)
HTTP Status Codes
Three-digit value. First digit = category.
CodeCategoryDescriptionExample
1xxInformationalRequest received, continuing processing100: Client should continue sending request
2xxSuccessAction successfully received, understood, accepted200: Request succeeded
3xxRedirectionFurther action must be taken to complete request301: Resource moved permanently to new URI

🧠 STATUS CODE TRICK
1=Info, 2=Success, 3=Redirect, 4=YOUR fault (client), 5=SERVER's fault. "4xx = you screwed up, 5xx = they screwed up"

User Agent Detection & Session Handling
USER AGENT DETECTION

Adaptive Content

Server reads User-Agent header to adapt delivered content to different devices.

Examples:
• "Mozilla (compatible; MSIE; Windows; .NET CLR)"
• "Mozilla [en] (Android)"
• "Mozilla (compatible; MSIE; Windows Mobile; PPC; 240x320)"

Server-side code checks userAgent string → returns different page type (HTML, mobile-adapted, XHTML). Same applies to Accept-Language for content language adaptation.

SESSION HANDLING

Stateful Web Apps

HTTP is stateless, but web apps need state/sessions. Solution: Session ID exchange.

1
Server sends session ID to browser.
2
Browser stores it as a Cookie (small data piece on client-side).
3
Browser includes session ID in later requests.
4
Server associates request with correct session data structure.
Famous technique = Cookies = small pieces of data stored client-side. Session data structure lives on the server.
Download of Associated Resources

After receiving HTML document, browser analyzes it for associated resources (CSS files, images, scripts) that must be downloaded for presentation.

<link rel="stylesheet" href="main.css"> ← triggers separate GET request
<img src="banner.jpg" /> ← triggers another GET request
Each associated resource = separate HTTP request. This is why HTTP/2 multiplexing is such a big deal!
HTTP/2

PERFORMANCE OPTIMIZED — APPROVED FEBRUARY 2015

Goal: Optimized transport for HTTP semantics
THE PROBLEM WITH HTTP 1.1

Not Optimized for Performance

HTTP 1.1 was optimized for simplicity and wide applicability. Performance was improved vs 1.0, but the protocol itself was NOT designed for performance.

HTTP/2 SOLUTION

Two Key Techniques

Multiplexing Header Compression
HTTP methods, status codes, and semantics are NOT modified by HTTP/2 — same semantics, better transport!
HOW HTTP/2 WORKS — FRAMES & CHANNELS
Two HTTP messages are split into a sequence of four interleaved "frames"
1
Frames: Each HTTP message is split into smaller frames (HEADER frame + DATA frame for body).
2
Interleaving: Frames from different messages are interleaved. Example: Message1-HEADER, Message2-HEADER, Message1-DATA, Message2-DATA all sent together.
3
Channels: Frames are embedded into a channel. Frames may be prioritized enabling faster delivery of selected frames.
4
Multiple channels per TCP: Per TCP connection, several parallel channels can be established = true multiplexing!
🧠 HTTP/2 ANALOGY
HTTP/1.1 = one-lane road (one car at a time). HTTP/2 = multi-lane highway with interchangeable lanes. Same destination rules, way more efficient travel.
Approved by IETF steering group for publication as standards-track RFCs in February 2015.
Web Browser

ARCHITECTURE — 8 COMPONENTS

Browser = WAY more than just "displays pages"
EVOLUTION

Early browsers: Only requested documents and presented them after rendering. Today: Extended with XML support, script engines, and plug-in engines.

WEB BROWSER REFERENCE ARCHITECTURE — ALL 8 COMPONENTS
1. User Interface
Front-end for displaying a page to the user. What you actually see and interact with.
2. Browser Engine
Embedded component providing a HIGH-LEVEL INTERFACE for querying and using the rendering engine. Intermediary between UI and rendering.
3. Rendering Engine
Performs PARSING and LAYOUT for (X)HTML documents enriched with CSS. Core function!
4. Networking
Realizes HTTP communication with the server.
5. XML Parser
Parses XML content.
6. Script Engine
Executes scripts (JavaScript) embedded in (X)HTML pages.
7. Display Backend
Provides drawing and windowing primitives, UI widgets and fonts. Example: GTK+ (GNU Image Manipulation Program Toolkit).
8. Data Persistence
Stores associated data — cached pages, cookies, etc.
Plug-in Engine: Dynamic extension point for plug-ins (e.g., Web feed reader plug-in). Not always counted in core 8.
Note: Further subsystems like XSLT processor are NOT shown in reference architecture.
🧠 MEMORY — 8 COMPONENTS
U B R N X S D P
User Interface → Browser Engine → Rendering Engine → Networking → XML Parser → Script Engine → Display Backend → Persistence (Data)

HTML5 Web Workers
PARALLEL PROCESSING IN BROWSER
Definition: JavaScript that runs in the BACKGROUND, independently of UI scripts → allows parallel processing.
Why important? Performing computationally expensive tasks WITHOUT interrupting the UI. No frozen browser!
Communication: Interaction between Web Workers and UI thread is EVENT-BASED.
1
Create worker: var worker = new Worker("counter_code.js") — creates and starts new Web Worker executing code in specified file.
2
Worker sends message: Inside worker file, postMessage(i) sends message back to website containing value. (Messages from website to worker: same way: worker.postMessage(value))
3
Handler executes: worker.onmessage = function(event){...} — event handler executed, value of counter displayed in website.
🧠 KEY DISTINCTION
Web Workers can't directly access the DOM. They communicate via messages. This is the key limitation to remember!
HTML5 Storage — 3 Types
Same-origin policy: Scripts can ONLY access data that a script from the SAME source (protocol + hostname + port) has stored. All three storage types apply this!
TypeDescriptionAdvantageDisadvantage
Web Storage Simple key/value persistence system Very simple API, supported by most browsers Only simple use cases. No query language, no schemas, no transactional safety
Indexed Database API (IndexedDB) API to operate on indexable object stores persisted in browser Simplifies programming model while offering powerful persistence layer Not implemented in all browsers yet
Web SQL Database Simplified SQL implementation embedded in browser Efficient + feature-rich SQL (joins, inner selects, etc.) Specification effort STOPPED by W3C due to lack of implementation!
Web Storage — Details
TWO SCOPES
Session Storage: Valid as long as the top-level browsing context (= usually window or tab) is active. NOT shared between tabs.
Local Storage: NO temporal restrictions. Data can be accessed from different windows or tabs. Persists after closing browser!
removeItem('itemKey') — remove single item
clear() — remove ALL items
API USAGE
// Session Storage
sessionStorage.setItem('myKey', 'myValue');
sessionStorage.getItem('myKey');

// Local Storage
localStorage.setItem('myKey', 'myValue');
localStorage.getItem('myKey');

// Storage Event Listener
window.addEventListener('storage',
  function(event) {
    event.key; event.oldValue;
    event.newValue;
  });
LocalStorage modification → Storage Event propagated to other tabs → can be handled via event listeners!
IndexedDB — API Details
INDEXED DATABASE API — 3 STEP FLOW
1
Open database:
var request = window.indexedDB.open("BookDB");
request.onsuccess = function(event){ var database = event.result; }
→ If successful, onsuccess event handler executes.
2
Create object store:
objectStore = database.createObjectStore("BookList", "id");
→ Creates object store named "BookList" with index "id".
3
Add data:
var book = {"bookName": "Web Apps", "price": 20, "rating": "excellent"};
var request = objectStore.add(book);
→ If successful, onsuccess handler executes and key can be read.
Database operations result in DOM events that must be handled. Key-, index-, and cursor-based access available.
