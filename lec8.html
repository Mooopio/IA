🌐1. The Web & Search Engine Architecture

Think of the web as an enormous, unorganized library where everyone can add books, change content, and write in any language. A search engine must automatically explore, organize, and rank all of this to answer your question in milliseconds.

Why Is the Web Challenging?

Unlike a traditional library with a card catalog, the web has no central authority. This creates 5 key challenges:

  • Billions of interconnected pages — scale is enormous
  • Anyone can publish — no quality control
  • Various formats & languages — HTML, PDF, video, Arabic, English…
  • A lot of low-quality and duplicate content
  • Dynamic content — pages generated by JavaScript on-the-fly
📌 Exam Tip
Search engines must bring semantics to a semantic-free structure. This phrase is often directly quoted in exams.

⚠️ Key Distinction
The Crawler + Page Repository + Indexing pipeline is query-independent — it runs continuously in the background. The Query + Ranking modules are query-dependent — triggered per user search.

🕷2. Web Crawling — Deep Dive

What is a Web Crawler?

A web crawler (also called a web robot or web scutter) is an automated program that starts from a seed set of URIs and systematically discovers and downloads web pages.

Analogy: Imagine an explorer starting in Cairo, then following every road sign to discover new cities, and documenting everything along the way — following new signs from each new city.

Universal Crawler Architecture — Step by Step

Step 1: Initialize the Frontier▼

The Frontier is a list of URIs waiting to be visited. It is initialized with Seed URIs — a starting set provided by the user or system. Think of it as the crawler's "to-do list."

Step 2: Dequeue URI from Frontier▼

The crawler picks the next URI from the frontier. The order depends on the crawling strategy (breadth-first or depth-first).

Step 3: Fetch the Page▼

Uses a regular HTTP client to download the page. Key efficiency improvements:

  • Timeout mechanism — avoid wasting time on slow/unresponsive servers
  • Efficient socket communication — better than default library classes
  • Parallelizing frontier access — multiple URIs downloaded simultaneously
  • DNS pre-fetching — resolve domain names to IPs ahead of time
  • Multithreaded download — multiple simultaneous downloads
Step 4: Extract URIs & Add to Frontier▼

Extract all <a href="..."> links from the page. Then canonicalize them:

  1. Convert relative URIs to absolute: /WORLD → http://www.cnn.com/WORLD
  2. Add/remove port numbers: http://cnn.com/ → http://cnn.com:80/
  3. Remove fragment identifiers: page.html#section3 → page.html

Only add to frontier if the URI is not already present (deduplication).

Step 5: Store Page & Check Termination▼

Store the fetched page in the Page Repository. Stop when:

  • A specified number of pages has been crawled, OR
  • The frontier is empty (no more URIs to visit)

Crawling Strategies: BFS vs DFS

Web Page Parsing & DOM

Raw HTML is often badly written (unclosed tags, missing attributes). Crawlers must:

  1. Tidy the HTML — transform malformed markup into well-formed XML
  2. Convert to DOM (Document Object Model) — a tree structure allowing easy programmatic access
  3. Use web scraping — techniques to extract data from this structure
📌 Definition to Memorize
Web Scraping = Accessing and extracting unstructured information from (X)HTML pages.

Focused Crawlers

General crawlers collect everything. Focused (topical) crawlers collect only pages matching specific criteria using a classification component:

  • Avoid certain top-level domains (e.g., no .com)
  • Require specific terms in the page content
  • Require a specific page structure (e.g., blog format)

Scraper sites are a common application — aggregating content from multiple sources on a specific topic.

🤖3. Robots Exclusion Standard & Sitemaps

Robots.txt — The Crawler's Rulebook

The Robots Exclusion Standard is a convention (not a real standard — defined by consensus) stored in a file called robots.txt at the website root (www.example.com/robots.txt).

Why site admins use it:

  • Privacy — prevent certain content from being indexed by search engines
  • Relevance — exclude irrelevant subpages from crawling
⚠️ Important Nuance
robots.txt only works with cooperative crawlers. Malicious bots (spam bots) ignore it entirely.

robots.txt Example Breakdown:

User-agent: * → Applies to ALL crawlers
Disallow: /cgi-bin/ → Block /cgi-bin/ directory
Disallow: /tmp/ → Block /tmp/ directory
Disallow: /private/ → Block /private/ directory

User-agent: Googlebot-Image → Only Google Image Search
Disallow: /fotos → Block photo directory for Google images

User-agent: yahoo-blogs/v3.9 → Only Yahoo blog crawler
Disallow: /blog → Block blog directory for Yahoo

Crawl-delay: 60 → Minimum 60 sec between visits
Sitemap: http://www.example.com/sitemap.xml.gz

Sitemaps Protocol

Problem: Some pages are only reachable via AJAX (XMLHttpRequest) — JavaScript loads links dynamically, which crawlers can't execute. Solution: Sitemaps — an XML file that explicitly lists all available URIs.

<urlset>
  <url>
    <loc>http://www.example.com/index.html</loc>
    <lastmod>1970-06-07</lastmod> ← Last modification date
    <changefreq>monthly</changefreq> ← How often it changes
    <priority>0.8</priority> ← Relative importance (0.0–1.0)
  </url>
</urlset>

📊4. Relevance Ranking & PageRank

Why Classical TF-IDF Falls Short on the Web

TF-IDF (Term Frequency × Inverse Document Frequency) works well for traditional document collections. On the web, it fails because:

  • Queries are very short (average: 2.35 terms)
  • Enormous variety of documents and vocabulary
  • Anyone can publish — misinformation is rampant
  • Content can be artificially manipulated (keyword stuffing)
Overall Score = Content Score (TF-IDF) + Structure Score (Popularity)

PageRank — Core Intuition

Invented by: Google founders Brin & Page (hence PageRank).

Core idea: A hyperlink from page A to page B is a recommendation of B by A's author. Quality is measured recursively:

  • A page is important if many pages link to it (high in-degree)
  • A page is even more important if the pages linking to it are themselves important
💡 Analogy
Academic citations work the same way. A research paper cited by Nobel Prize winners is more valuable than one cited by unknown blogs — even if both have 10 citations.

The Random Walk Model

PageRank models an average web user randomly clicking links:

  • At each page with N outgoing links, move to one successor with probability 1/N
  • This distributes the page's "prestige" equally: each link gets Prestige(current) / N
  • Popularity = how often the random user visits a page over time

Iterative PageRank Calculation

Initialization: Assign all pages a prestige of 1 or 1/N.

Each iteration: Prestige(p) = Σ [ Prestige(q) / outdegree(q) ] for all q that link to p.

📌 Exam Formula
Prestige(p) = Σ [ Prestige(q) / outdegree(q) ]
for all nodes q where (q → p) exists
This is applied iteratively until values converge.

Problems with Simple Random Walk

ProblemDescriptionEffect
Cyclic GraphsUser gets trapped in a loop (A→B→A→B…)Prestige concentrates in the cycle, starving other nodes
Dangling NodesA page with NO outgoing linksPrestige is absorbed and never redistributed — "prestige sink"

The Extended Random Walk — PageRank Formula

The solution: add a damping factor d — the probability that a random user enters an arbitrary new URI instead of clicking a link.

PageRank(p) = d/|T| + (1-d) × Σ [ PageRank(q) / outdegree(q) ]

Where:
d = probability of entering a new URI (teleportation)
|T| = total number of nodes in the graph
(1-d) = probability of clicking an outgoing link
q = any node with a link to p
outdegree(q) = number of outgoing links from q

This formula can also be expressed as an eigenvector problem or a linear homogeneous system.

🔗5. HITS — Hubs & Authorities

Two Types of Important Pages

🏛 Authority

A page with many in-links. It has good content on a topic, so many people trust it and point to it.

Example: Wikipedia article on Machine Learning

High in-degree

🗺 Hub

A page with many out-links. It serves as an organizer/directory, pointing to many good authorities.

Example: A curated "Best ML Resources" page

High out-degree

Mutual Reinforcement (The Circular Thesis)

📌 Core HITS Principle — Must Memorize
"A page is a good hub if it points to good authorities. A page is a good authority if it is pointed to by good hubs."
Repeat until convergence:
HUB[x] = Σ AUTH[r] for all r where (x → r) exists
AUTH[x] = Σ HUB[q] for all q where (q → x) exists

BasisIn-degree (weighted by quality)Both in-degree (auth) and out-degree (hub)

📋Smart Summary

High-yield bullet-point summary — every point is exam-relevant.

🌐 Search Engine Architecture▼
  • Web = semantic-free structure; search engines add semantics
  • 4 modules: Crawler → Page Repository → Indexing → Indexes (query-independent)
  • + Query Module → Ranking Module (query-dependent)
  • Two index types: Content Index + Structure Index
🕷 Web Crawling▼
  • Crawler = web robot / web scutter
  • Starts from Seed URIs → maintains a Frontier (unvisited URIs list)
  • Termination: frontier empty OR page count limit reached
  • Fetching efficiency: timeouts, socket optimization, parallelism, DNS pre-fetch, multithreading
  • BFS = queue-based, level-by-level | DFS = stack-based, path-by-path with backtracking
  • Link extraction: find <a href> tags → filter → canonicalize → add to frontier if new
  • Canonicalization: relative→absolute, add/remove ports, remove fragments
  • Web Scraping = extracting unstructured data from (X)HTML
  • DOM representation = tree structure for easy page access
  • Tidying = converting malformed HTML to well-formed XML before parsing
  • Focused/Topical crawlers use a classification component to target specific content
🤖 Robots.txt & Sitemaps▼
  • robots.txt = convention (not standard) at site root for cooperative crawlers
  • Directives: User-agent, Disallow, Crawl-delay, Sitemap
  • Only cooperative crawlers respect it — malicious bots ignore it
  • ACAP = Automated Content Access Protocol (extension/improvement)
  • Sitemaps = XML protocol to list URIs not reachable via normal links (e.g., AJAX pages)
  • Sitemap fields: loc (URL), lastmod, changefreq, priority (0.0–1.0)
📊 Relevance Ranking & PageRank▼
  • Classical ranking: TF-IDF — insufficient for web (short queries, low quality content)
  • Web ranking = Content Score + Structure/Popularity Score
  • PageRank: invented by Brin & Page; hyperlink = recommendation
  • Quality = in-degree + quality of linking pages (recursive)
  • Random Walk Model: user moves to successor with probability 1/N
  • Prestige is distributed as Prestige(p)/N to each outgoing link
  • Calculation: iterative, starts with all pages = 1 or 1/|T|
  • Problems: cyclic graphs trap prestige; dangling nodes absorb prestige
  • Fix: damping factor d — probability of teleporting to random page
  • Formula: PR(p) = d/|T| + (1-d) × Σ[PR(q)/outdegree(q)]
  • Also solvable as eigenvector problem or linear system
🔗 HITS — Hubs & Authorities▼
  • HITS = Hyperlinked Induced Topic Search
  • Authority = many in-links (good content) | Hub = many out-links (good organizer)
  • Mutual reinforcement: good hubs point to good authorities and vice versa
  • HITS is query-dependent: starts with initial result set for a query
  • Produces two ranked lists: one for hubs, one for authorities
  • Iterative convergence: HUB[x] = Σ AUTH[r]; AUTH[x] = Σ HUB[q]

Common Exam Traps & Mistakes

❌ Trap 1: "Robots.txt is a standard"
robots.txt is a convention defined by consensus, NOT an official standard. It also only applies to cooperative crawlers.
❌ Trap 2: "PageRank is query-dependent"
PageRank is query-independent (precomputed). HITS is query-dependent.
❌ Trap 3: "Hubs have many in-links"
Hubs have many out-links. Authorities have many in-links. Don't mix these up!
❌ Trap 4: "Removing fragment identifiers changes the page"
Fragment identifiers (#section3) only affect browser scrolling position — removing them during canonicalization does NOT change the actual page downloaded.
❌ Trap 5: "BFS uses a stack"
BFS uses a Queue (FIFO). DFS uses a Stack (LIFO).
❌ Trap 6: "Dangling nodes distribute prestige to all"
Dangling nodes (no outgoing links) absorb prestige — they never redistribute it. That's the problem the damping factor solves.
