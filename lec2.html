Web Application Architecture — The Big Picture
الصورة الكبيرة — إيه اللي بيحصل لما بتفتح موقع؟
تخيل إنك بتطلب أكل أونلاين. أنت (Client) بتبعت الطلب، في وسيط (HTTP) هو البريد اللي بيوصل الطلب، والمطعم (Server) هو اللي بيجهز الأكل ويرجعه ليك.

في الـ Web Application في 3 طبقات (Tiers):
🔵 Presentation (العرض): اللي بتشوفه على الشاشة — HTML, CSS
🟡 Logic (المنطق): الكود اللي بيشتغل ويعمل حسابات — JavaScript على الكلاينت أو Java/PHP على السيرفر
🟢 Data (البيانات): قاعدة البيانات — Database

الكلاينت (المتصفح) بيبعت Request، السيرفر بيعملها Process، وبيرجع Response. ده هو الدورة كلها.
💡
لاحظ: في الـ Slide 5 بتوضح إن الـ Web Applications اتطورت — زمان كانت كل اللوجيك على السيرفر، دلوقتي جزء كبير بقى على الكلاينت (JavaScript). ده اللي بيخلي المواقع أسرع وأذكى.
01
REST Model
Representational State Transfer
🔄
REST = Representational State Transfer
نموذج REST — إيه هو بالظبط؟
REST ده نموذج (Model) للتواصل بين الأنظمة المتوزعة على الإنترنت. مش تقنية ولا لغة — ده أسلوب تفكير.

🔑 الفكرة الأساسية: بدل ما تنادي على Function أو Procedure زي RPC، أنت بتتعامل مع Resources (موارد/وثائق). كل حاجة ليها عنوان واضح (URI).

مثال: بدل ما تقول "اجيبلي بيانات المستخدم رقم 5" بالكود، أنت بتعمل GET على /users/5 وبتاخد البيانات.
⚠️
حاجة مهمة جداً: REST بيقول إن البروتوكول Stateless — يعني كل Request لازم يكون فيها كل المعلومات اللازمة. السيرفر مش بيفتكر حاجة من Request للتانية. ده مختلف عن الجلسة التقليدية.
🆚
REST vs RPC: في RPC أنت بتنادي على function زي getUser(5). في REST أنت بتعنون resource زي GET /users/5. الفرق إن REST بيستخدم HTTP methods كـ interface موحد للعالم.
خاصية REST شرح مثال
Client/Server الكلاينت والسيرفر منفصلين المتصفح ≠ السيرفر
Stateless كل Request مستقلة — مفيش ذاكرة كل call لازم تبعت التوكن فيها
Resource-based بتتعامل مع موارد مش إجراءات GET /products/12
Uniform Interface GET, POST, PUT, DELETE — نفس اللغة أي لغة تقدر تتكلم مع أي سيرفر
02
Web Server Architecture
معمار الـ Web Server من الداخل
🖥
Web Server = Software that delivers resources via HTTP
الـ Web Server إيه هو بالظبط؟
الـ Web Server ده Software (برنامج) بيشتغل على جهاز وبيخلي الـ Resources (ملفات، صفحات) متاحة عبر HTTP.

لو المحتوى Static (ثابت) زي HTML files: السيرفر بس بياخد الملف من الـ File System ويبعته.

لو المحتوى Dynamic (متغير): السيرفر بيشغّل Logic (كود) ويولّد المحتوى لحظياً. ده اللي بيحصل في مواقع زي فيسبوك — صفحتك مش ملف ثابت، بتتبنى كل مرة.
①
Startup — بداية السيرفر

لما السيرفر بيشتغل، بيعمل على طول "Processing Pool" — يعني بيجهّز مجموعة من الـ Processing Entities مسبقاً علشان يكون جاهز. ده أسرع من إنه ينشئ واحدة كل شغلة.

②
Connection — المتصفح بيتصل

المتصفح بيعمل TCP Connection مع السيرفر من خلال interface معين (Port). ده بيتعمل عبر الـ Connector Entity.

③
Client Descriptor — بيانات الاتصال

الـ Connector Entity بيعمل "Client Descriptor" — ده زي بيانات الاتصال بالعميل. بعدين بيبعته للـ Delegation Process.

④
Delegation — التوزيع

الـ Delegation Process بياخد الـ Client Descriptor ويديه لـ Processing Entity فاضية من الـ Pool.

⑤
HTTP Communication — التواصل الفعلي

دلوقتي الـ Processing Entity بتتكلم مع المتصفح مباشرة عبر HTTP. ده بيسمح بـ Parallel Processing — أكتر من متصفح في نفس الوقت.

✅
ليه Pool وليه مش ننشئ Entity كل مرة؟ إنشاء Thread أو Process من الصفر بياخد وقت. لو عندنا 1000 Request في نفس الوقت وبننشئ Thread لكل واحدة — السيرفر هيانهار. الـ Pool بيحل المشكلة دي بإن الـ Entities جاهزة.
⚙️
Server-side Logic — CGI, Servlet, Application Server
اللوجيك على السيرفر — إزاي بيتعمل المحتوى الديناميكي؟
المحتوى الديناميكي بيتعمل بـ 3 طرق رئيسية:

🔹 CGI (Common Gateway Interface): أقدم طريقة. لكل Request السيرفر بيشغّل Script جديد. بطيء جداً لكن بسيط.
🔹 Java Servlet Container: الـ Servlet بيشتغل جوه الـ Server مباشرةً. أسرع من CGI.
🔹 Application Server: الأقوى — بيوفر Services إضافية زي: Transaction Management, Security, Directory Services. مناسب للأنظمة الكبيرة.

بعدين في حاجة اسمها (X)HTML Engine — ده بياخد المحتوى الديناميكي ويدمجه في Templates HTML جاهزة. زي إنك بتعمل "قالب" للصفحة وبتملاه ببيانات.

04
HTTP Protocol
الـ HTTP 1.1 بالتفصيل الممل — كل حاجة هتيجي في الامتحان
📡
HTTP = HyperText Transfer Protocol (runs on top of TCP)
HTTP — البروتوكول اللي بيحكم كل حاجة على الإنترنت
HTTP ده بروتوكول Request-Response — يعني أنت بتطلب وهو بيرد. بيشتغل فوق TCP.

كل Message (Request أو Response) بتنقسم لـ جزئين:
📋 Header: المعلومات — إيه اللي أنت عايزه، مين أنت، إيه اللي مقبوله
📦 Body: المحتوى الفعلي — الملف، البيانات، HTML

الـ Resources بتتعنون بـ URI (Uniform Resource Identifier):
• Absolute URI: http://www.example.com/index.html
• Absolute Path: /index.html
• Asterisk *: لما مش في resource محدد
🔑
CRLF مهم جداً: الـ Header والـ Body بيتفصلوا بـ CRLF (Carriage Return + Line Feed = \r\n). السطر الفاضي (CRLF وحده) هو اللي بيقول للسيرفر "الـ Header خلص، الـ Body بيبدأ هنا".
🔗
HTTP 1.0 vs HTTP 1.1 — Persistent Connections
الفرق بين 1.0 و 1.1 — التحسين الكبير
في HTTP 1.0: كل Request بتعمل TCP Connection جديدة وبتقفلها. ده زي إنك كل ما تعمل أوردر تتصل بالمطعم وبعد ما يردوا تقفل التليفون. بطيء جداً!

في HTTP 1.1: بقى في Default الاتصال يفضل مفتوح — Connection: keep-alive. أنت بتعمل Request وتاني وتالت على نفس الـ TCP Connection. أسرع بكتير!

❌ HTTP 1.0 — قديم وبطيء

  • كل Request → TCP Connection جديدة
  • 226 packets من الكلاينت
  • 271 packets من السيرفر
  • إجمالي: 497 packets
  • overhead كبير جداً

✅ HTTP 1.1 — أسرع

  • Persistent Connection by default
  • 70 packets من الكلاينت
  • 153 packets من السيرفر
  • إجمالي: 223 packets
  • Pipeline: 83 packets فقط!
🚀
HTTP 1.1 Pipelining — Send requests without waiting
الـ Pipelining — إرسل من غير ما تستنى الرد
الـ Pipelining ده تقنية بتخليك تبعت Request 2 و Request 3 قبل ما تستقبل الرد على Request 1.

⚠️ تحذير مهم: الـ Responses لازم ترجع بنفس الترتيب اللي الـ Requests اتبعت بيه. لو Request 1 اتأخرت، Request 2 و3 بتستنى — ده اتسمى Head-of-Line Blocking.

🚨 مش كل المتصفحات بتدعمه — ده ضعف في الـ Pipelining.

كمان في الـ HTTP 1.1، رسايل HTTP متعددة ممكن تتبعت في نفس TCP Segment واحدة. ده بيوفر bandwidth.
🛠
HTTP 1.1 Methods — GET, POST, PUT, DELETE, etc.
الـ HTTP Methods — إيه الفرق بينهم بالظبط؟
الـ Methods بتحدد إيه اللي عايزه من السيرفر. النجمة (*) بتتبعت لو مش في Resource محدد.
Methodالوظيفةفي HTTP 1.0؟مثال
GET اجيبلي الـ Resource ✅ نعم GET /index.html
HEAD زي GET بس من غير Body — بس Header ✅ نعم لمعرفة حجم الملف قبل تحميله
POST ابعت بيانات للـ Resource ✅ نعم Submit form, Login
PUT * اعمل Resource جديد أو عدّله ❌ 1.1 فقط Upload ملف
DELETE * امسح الـ Resource ❌ 1.1 فقط حذف post
OPTIONS * إيه الـ Methods اللي السيرفر بيدعمها؟ ❌ 1.1 فقط CORS Pre-flight
TRACE * السيرفر بيرجع الـ Request تاني — للتشخيص ❌ 1.1 فقط Debugging
CONNECT * محجوز للـ Proxy Servers ❌ 1.1 فقط HTTPS tunneling
⚠️
HEAD method — سؤال امتحان شائع: HEAD هو نفس GET بالظبط، بس الرد بيجي من غير Message Body. مفيد لو عايز تعرف حجم الملف أو نوعه من غير ما تنزله كله.
🔢
HTTP Status Codes — 1xx, 2xx, 3xx, 4xx, 5xx
الـ Status Codes — رسايل السيرفر
الـ Status Code ده 3 أرقام — الرقم الأول بيحدد الفئة.
الكودالفئةالمعنىأمثلة مهمة
1xx Informational الطلب اتستقبل، استمر 100 Continue
2xx Success تمام، نجح! 200 OK
3xx Redirection الـ Resource اتنقل 301 Moved Permanently
4xx Client Error غلطة من عندك أنت 404 Not Found
5xx Server Error غلطة من عند السيرفر 501 Not Implemented

HTTP Communication Details — Real Examples
مثال حقيقي على Request و Response
دي أمثلة حقيقية من الـ Slides — ادرسهم كويس!

مثال 1: HEAD Request (لو الملف مش موجود)

/* REQUEST */ HEAD /folder/indx.html HTTP/1.1\r\n ← typo! اسم الملف غلط Host: www.webmailer.ws\r\n Connection: close\r\n Accept-Encoding: gzip\r\n ← بيقول: أنا باقبل gzip Accept: text/xml,application/xml,text/html,*/*\r\n Accept-Language: en-gb\r\n User-Agent: Mozilla\r\n \r\n ← CRLF = نهاية الـ Header /* RESPONSE */ HTTP/1.1 404 Not Found\r\n ← Status Code Server: Apache (Unix) PHP\r\n Content-Type: text/html\r\n Content-Length: 0\r\n ← HEAD = Body فاضي Date: Fri, 12 Jan 14:25:15 GMT\r\n Connection: close\r\n \r\n

مثال 2: POST Login (بيانات الـ Form في الـ Body)

/* REQUEST */ POST /cgi/login HTTP/1.1\r\n Host: www.webmailer.ws\r\n Connection: Keep-Alive\r\n ← Persistent Connection Content-type: application/x-www-form-urlencoded\r\n Content-length: 41\r\n \r\n login=me@webmailer.ws&password=pass123 ← البيانات في الـ Body /* RESPONSE */ HTTP/1.1 200 OK\r\n Content-Encoding: gzip\r\n ← الرد متضغوط بـ gzip Content-Length: 1922\r\n Keep-Alive: timeout=5, max=1024\r\n ← Connection تفضل 5 ثواني، max 1024 request \r\n ...gzip encoded HTML content...
🧠
لاحظ الفرق: في POST الـ Form data بتتبعت في الـ Message Body. في GET الـ data بتتبعت في الـ URL نفسه (?name=value&name2=value2). ولكن GET مش مناسب لكلمات السر!
📥
HTTP Download of Associated Resources
إزاي المتصفح بيجيب الـ CSS والصور؟
لما تطلب صفحة HTML، السيرفر بيبعت الـ HTML بس. المتصفح بعدين بيعمل Parse للـ HTML ويشوف فيها إيه — CSS، صور، scripts. وبعدين بيبعت Requests منفصلة لكل واحدة.

مثلاً:
① GET /index.html → يستقبل الـ HTML
② بيشوف في الـ HTML: <link rel="stylesheet" href="main.css"> و <img src="banner.jpg">
③ بيبعت Request لـ main.css
④ بيبعت Request لـ banner.jpg
⚡
Performance impact: لو صفحة فيها 50 صورة → 50 HTTP Request إضافية! عشان كده HTTP/2 جاء بالـ Multiplexing.
📱
HTTP User Agent Detection — Adapt content to device
اكتشاف نوع الجهاز — User-Agent Header
كل Browser بيبعت في الـ Request Header حاجة اسمها User-Agent — دي string بيعرّف المتصفح والنظام.

السيرفر بيقرأ الـ User-Agent ويبعت نوع مختلف من المحتوى. أمثلة:
• Mozilla (compatible; MSIE; Windows; .NET CLR) → Windows IE
• Mozilla [en] (Android) → Android phone
• Mozilla (compatible; MSIE; Windows Mobile; PPC; 240x320) → Mobile 240×320px
// Server-side code (Java) String userAgent = request.getUserAgent(); if (userAgent.contains("Android")) pageType = HTML; ← Android phone → full HTML else if (userAgent.contains("240x320")) pageType = 240_320_ADAPTED; ← old mobile → adapted version else pageType = XHTML; ← default
🌐
نفس الفكرة مع اللغة: Accept-Language: en-gb → السيرفر ممكن يبعت الصفحة بالإنجليزي البريطاني. الـ HTTP Header بيدي معلومات كتير عن الكلاينت.
05
Session Handling
إزاي السيرفر بيتعرف عليك بين الـ Requests؟
🔐
Sessions + Cookies — State management over stateless HTTP
الجلسات والكوكيز — إزاي المواقع بتفتكرك؟
HTTP هو Stateless — يعني السيرفر ما بيفتكرش أي حاجة بين الـ Requests. إذن إزاي فيسبوك عارف إنك لسه مسجّل دخول؟

الحل: Sessions!
① أنت بتعمل Login → السيرفر بيعمل Session Data Structure وبيديلها ID فريد (مثلاً: 1f2e22f467ab1)
② السيرفر بيبعت الـ Session ID للمتصفح عبر Cookie
③ في كل Request جديدة، المتصفح بيبعت الـ Cookie أوتوماتيكياً
④ السيرفر بيقرأ الـ Session ID من الـ Cookie ويعرف مين أنت

الـ Cookie ده ببساطة بيانات صغيرة مخزّنة على الجهاز بتاعك.
06
HTTP/2
التطور القادم — Multiplexing + Header Compression
⚡
HTTP/2 — Approved Feb 2015, Multiplexing + Binary Frames
HTTP/2 — ليه أسرع بكتير؟
HTTP 1.1 كان كويس بس مش optimal. المشكلة: الـ Methods والـ Status Codes نفسها، بس الـ Transport مش كفء.

HTTP/2 هدفه: نفس الـ Semantics (GET, POST, 200, 404) بس Transport أحسن بكتير.

🔑 أهم التحسينات:
① Binary Protocol بدل Text — أسرع في Processing
② Multiplexing: أكتر من Stream في نفس TCP Connection — مفيش Head-of-Line Blocking
③ Header Compression (HPACK): الـ Headers بتتضغط
④ Server Push: السيرفر يبعت Resources قبل ما تطلبها
⑤ Stream Prioritization: تحديد أولوية الـ Streams

الـ Frames هي وحدة البيانات في HTTP/2 — كل Message بتتقسم لـ Frames، ممكن تتخلط مع Frames من Messages تانية على نفس الـ Connection.
✅
الـ Methods والـ Status Codes مش اتغيروا: GET لسه GET، 404 لسه 404. HTTP/2 بس غيّر إزاي البيانات بتتنقل، مش إيه اللي بيتنقل. ده مهم جداً!

Web Workers — Background JavaScript Threads
الـ Web Workers — JavaScript في الخلفية
المشكلة: JavaScript بتشتغل في Thread واحدة. لو عندك حسابات كبيرة، الـ UI بيتجمد!

الحل: Web Workers — JavaScript بيشتغل في Background Thread مستقلة، ومش بيأثر على الـ UI.

🔑 التواصل Event-based:
• الـ Main Thread بيبعت: worker.postMessage(value)
• الـ Worker بيبعت: postMessage(i)
• الـ Main Thread بيستقبل: worker.onmessage = function(event){...}

⚠️ الـ Web Worker مش قادر يلمس الـ DOM — مش يعدّل الصفحة مباشرة. بيبعت النتيجة للـ Main Thread وهي اللي بتعدّل.
/* ===== Main Page (website) ===== */ // 1. Create & start Web Worker var worker = new Worker("counter_code.js"); // 3. Receive message from Worker worker.onmessage = function(event) { document.getElementById("result").innerHTML = event.data; }; /* ===== counter_code.js (Web Worker) ===== */ var i = 0; function timedCount() { i = i + 1; postMessage(i); // 2. Send to main page setTimeout("timedCount()", 500); // every 500ms } timedCount();

// Storage Event — عارف لما localStorage بيتغير من Tab تانية window.addEventListener('storage', function(event) { console.log('Value for ' + event.key + ' changed from ' + event.oldValue + ' to ' + event.newValue); });

الحاجات الصعبة والفنيات اللي بتلغبط — ركّز هنا!

① REST Stateless — الفخ الأكبر

الخطأ الشائع: الناس بتقول "REST مش بيخزّن Session" وبعدين تتساءل "طيب إزاي موقع زي أمازون بيفتكرك؟". الإجابة: REST Stateless على مستوى البروتوكول — بس التطبيق ممكن يعمل State عبر Tokens أو Cookies في كل Request. الفرق إن السيرفر مش محتاج يخزن Session — الـ State بتتبعت في كل Request.

② HEAD Method — مش معروف كفاية

HEAD = GET بالظبط بس الرد بييجي من غير Body. فايدته: تعرف حجم الملف (Content-Length) ونوعه (Content-Type) قبل ما تنزله. مفيد جداً في Download Managers. لاحظ في المثال: Content-Length: 0 — حتى لو في Body، مش بيتبعتش في HEAD.

③ Pipelining ≠ Multiplexing

Pipelining في HTTP 1.1: بتبعت Requests من غير ما تستنى، بس الـ Responses لازم ترجع بنفس الترتيب (Head-of-Line Blocking). Multiplexing في HTTP/2: كل Request ليها Stream مستقلة — مفيش ترتيب ملزم، ومفيش Blocking. ده أهم فرق بينهم!

④ HTTP/2 مش غيّر الـ Semantics

HTTP/2 غيّر فقط Transport Layer — الـ Framing والـ Binary Encoding. الـ Methods (GET, POST...) والـ Status Codes (200, 404...) والـ Headers نفسها. بعض الناس بتفتكر إن HTTP/2 طريقة تانية للـ REST — أبداً، ده فقط Network Transport Optimization.

⑤ CRLF في HTTP — محتاج ينبه

كل سطر في الـ HTTP Message بينتهي بـ \r\n (CRLF). الـ Header بينتهي بـ \r\n\r\n (سطر فاضي). لو نسيت ده في امتحان شفهي أو عملي، بيبان إنك مش فاهم البروتوكول.

⑥ Web Storage Same-Origin Policy

sessionStorage وlocalStorage بيطبقوا Same-Origin Policy: Protocol + Hostname + Port لازم يكونوا نفس بعض. يعني https://example.com مش ممكن يوصل لـ Storage بتاع http://example.com — الـ Protocol مختلف!

⑦ Web Worker مش قادر يلمس DOM

Web Worker يشتغل في Thread منفصلة ومعزولة. مش قادر يعمل document.getElementById أو أي حاجة تخص الـ UI. بيتواصل مع الـ Main Thread فقط عبر Messages. لو حاول يلمس DOM → Error.

⑧ Connector Entity vs Delegation Process vs Processing Entity

3 كيانات مختلفة في معمار الـ Web Server:
• Connector: بيفتح الـ TCP Port ويستقبل الاتصالات، بيعمل Client Descriptor
• Delegation: بياخد الـ Client Descriptor ويوزعه على Processing Entity
• Processing Entity: بتعمل الشغل الفعلي — HTTP Communication
الـ Pool بيتعمل مسبقاً قبل أي Request.

⑨ Web SQL Database — مهمة جداً في الامتحانات

Web SQL Database هو standard اتوقف تطويره من W3C — مش لأنه مش كويس، لكن لأن مفيش implementations كافية. ده مختلف عن الـ IndexedDB اللي لسه active. السؤال المكتوب في الـ Slide: "Specification effort has been stopped by W3C due to lack of implementation" — احفظ الصيغة دي.

📋 EXAM SUMMARY — Everything You Need to Pass

🔄 REST Model

  • Representational State Transfer
  • Client/Server + Stateless protocol
  • Each message = all context info
  • Requests target resources, not procedures
  • Every resource = unique URI
  • Unlike RPC (which calls procedures)
  • Uses generic interface with standard semantics

🖥 Web Server

  • Software making resources available via HTTP
  • Hierarchical structure for parallel processing
  • ① Pool created at startup
  • ② Browser initiates TCP (Connector)
  • ③ Connector → Client Descriptor
  • ④ Delegation → Processing Entity
  • ⑤ Processing Entity does HTTP comms
  • Server-side: CGI / Servlet / App Server
  • (X)HTML Engine combines logic + templates

🌍 Browser Architecture

  • User Interface: front-end display
  • Browser Engine: high-level interface to renderer
  • Rendering Engine: parses HTML+CSS, does layout
  • Networking: HTTP communication
  • Script Engine: executes JavaScript
  • XML Parser: parses XML content
  • Display Backend: drawing primitives (GTK+)
  • Data Persistence: cache, cookies
  • Plug-in Engine: dynamic extensions

📡 HTTP Basics

  • Runs on top of TCP
  • Request-Response pattern
  • Every message = Header + Body (separated by CRLF)
  • Resources addressed by URI
  • URI forms: *, absolute URI, absolute path
  • Request Line: Method URI HTTP/version
  • Status Line: HTTP/version code text
  • Key-value pairs in headers

🔗 HTTP 1.0 vs 1.1

  • 1.0: new TCP per request → expensive
  • 1.1: Persistent Connection by default
  • 1.1: Connection: keep-alive
  • 1.1: Pipelining — send without waiting
  • Pipelining: responses in same order as requests
  • Pipelining: not supported by all browsers
  • Packet counts: 497 (1.0) → 223 (1.1) → 83 (pipeline)
  • Multiple HTTP messages in 1 TCP segment possible

🛠 HTTP Methods

  • GET: request resource delivery
  • HEAD: like GET but no body (meta only)
  • POST: submit data in body
  • PUT*: store content at URL
  • DELETE*: delete resource
  • OPTIONS*: get server's supported methods
  • TRACE*: echo request (diagnostic)
  • CONNECT*: reserved for proxies
  • * = not in HTTP 1.0, added in 1.1

🔢 Status Codes

  • 1xx: Informational — 100 Continue
  • 2xx: Success — 200 OK
  • 3xx: Redirection — 301 Moved Permanently
  • 4xx: Client Error — 404 Not Found
  • 5xx: Server Error — 501 Not Implemented
  • First digit = category
  • Numeric + textual representation

🔐 Sessions & Cookies

  • HTTP is stateless → need sessions for state
  • Server creates Session Data Structure on login
  • Server sends Session ID to browser via Cookie
  • Browser sends Cookie in every subsequent request
  • Cookie: small client-side data piece
  • Cookies scoped by source domain
  • Server maps sid → session data structure

⚡ HTTP/2

  • Goal: optimized transport, same semantics
  • Methods & status codes unchanged
  • Approved by IETF: February 2015
  • Binary protocol (not text)
  • Multiplexing: multiple streams per TCP connection
  • Header Compression (HPACK)
  • Messages split into interleaved Frames
  • Frames can be prioritized
  • Several parallel channels per TCP connection

🧩 HTML5 Overview

  • = Enhanced HTML + CSS3 + JavaScript APIs
  • Doctype: <!doctype html>
  • Addresses mobile & desktop needs
  • New structural elements for semantics
  • Built-in audio/video (no Flash needed)
  • Web Workers, Web Storage, Geolocation API
  • Aggregates previously separated standards

🏗 HTML5 Semantic Elements

  • <header>: introductory/navigational group
  • <nav>: navigation section
  • <section>: generic grouping section
  • <article>: independent content (blog, news)
  • <aside>: slightly related content (sidebar)
  • <footer>: author/copyright info
  • For grouping, NOT positioning (CSS does that)

⚙️ Web Workers

  • JS running in background, independent of UI
  • For expensive tasks without blocking UI
  • Interaction: event-based messaging
  • Main → Worker: worker.postMessage(val)
  • Worker → Main: postMessage(val)
  • Main receives: worker.onmessage
  • Worker CANNOT access DOM directly

💾 HTML5 Storage

  • All 3 apply Same-Origin Policy
  • Web Storage: simple key-value, most supported
  • sessionStorage: lives while tab is open
  • localStorage: persistent, shared across tabs
  • localStorage change → Storage Event fires
  • IndexedDB: powerful, indexes, cursors
  • Web SQL: stopped by W3C (lack of implementation)
  • removeItem(key) / clear() for cleanup

🔍 User Agent & Resources

  • User-Agent header identifies browser/device
  • Server adapts content based on User-Agent
  • Also adapts via Accept-Language
  • HTML only → browser parses for linked resources
  • Sends separate requests for CSS, images, JS
  • IndexedDB operations return DOM Events
  • onsuccess handler for async DB operations
