Phase 1 — Deep Understanding

Interaction Models in the WWW

A complete, deeply explained guide to how web applications communicate — from classic synchronous clicks to real-time bidirectional WebSockets.

What is Interaction?

Interaction on the web involves two layers of communication: the user interacting with an application (e.g., clicking a button), and the client (browser) communicating with the server (exchanging data). Both layers can work in very different ways depending on the model used.

Layer 1

User ↔ Application

The user performs actions (clicks, typing) that trigger the application to respond. The key question: is the user blocked while waiting for a response?

Layer 2

Client ↔ Server

Data is exchanged between the browser and web server. The key question: does every update require a new HTTP request?

Model 1 — Synchronous (Classic Web)

Click, Wait, and Refresh

This is the oldest model. Think of it like a vending machine: you insert a coin (click), the machine processes it (server works), and a product drops out (full page reloads). You can't do anything while you wait.

Definition: In the synchronous model, every user action that needs data triggers a full HTTP Request. The user is blocked (cannot interact with the page) until a complete new HTML page is returned and rendered.

How it works step-by-step

1

User clicks

The browser generates a full HTTP Request to the web server.

2

Server processes

The server handles the request (queries a database, runs logic) and generates a complete HTML page response.

3

Browser refreshes

The entire page is replaced with the new (X)HTML page. All context (scroll position, input state) is lost.

Shortcomings

  • Slow performance — the "click, wait, refresh" cycle adds noticeable latency
  • Loss of context — full page refresh destroys scroll position, form state, etc.
  • Excessive server load — entire pages are sent even if only one field changed
  • Low interactivity — user is blocked during every server round-trip

Model 2 — Non-Blocking (AJAX)

Asynchronous JavaScript + XML

AJAX is like a waiter at a restaurant. You place an order (user action), the waiter takes it to the kitchen (AJAX engine sends HTTP request), and you can keep talking to your friends (user continues interacting) while the food is being prepared. The waiter brings results back without you needing to go to the kitchen yourself.

History of AJAX

1999
Microsoft's "Outlook Web Access" was the first AJAX-like app using the XMLHttpRequest approach. Only IE supported it.
2002
Mozilla and Netscape implemented native XMLHttpRequest in their browsers (Mozilla 1.0, Netscape 7), but it didn't gain traction.
2005
Jesse James Garrett coined "AJAX" in an article. Google Suggest and Google Maps demonstrated its power → mass popularity.

AJAX Definition (J.J. Garrett)

"AJAX isn't a technology. It's really several technologies, each flourishing in its own right, coming together in powerful new ways."

XHTML + CSS (presentation)
DOM (dynamic display)
XML + XSLT (data interchange)
XMLHttpRequest (async retrieval)
JavaScript (glue)

The AJAX Engine — Key Concept

AJAX introduces an intermediary engine in the browser between the user interface and the server. The engine:

  • Intercepts user actions as JavaScript calls (not direct HTTP requests)
  • Can handle simple tasks locally (data validation, memory edits) without contacting the server
  • When server data is needed, it sends an HTTP Request asynchronously — the user is not blocked
  • Receives XML or text data (not full HTML pages) and updates only the relevant part of the DOM

XMLHttpRequest Object — The Heart of AJAX

The XMLHttpRequest object provides a client-side API for transferring data behind the scenes without a full page refresh.

IE Compatibility — Important Detail

  • IE5/IE6: used ActiveXObject("Msxml2.XMLHTTP")
  • Older IE: used ActiveXObject("Microsoft.XMLHTTP")
  • IE7+: supports native XMLHttpRequest

JSON — Alternative to XML

The X in AJAX stands for XML, but XML creates unnecessary overhead. JSON (JavaScript Object Notation) is the preferred modern alternative.

JSON advantages: Lighter/more compact than XML · Faster parsing (native JS data structures) · Language-independent but uses C-family conventions

JSON supports: Objects {}, Arrays [], Strings "", Numbers, Booleans (true/false), and nested structures.

AJAX Limitation: Although user actions are async, there is still a high degree of dependence between HTTP Request and Response — every server update still requires a full request/response round-trip from the AJAX engine.

Model 3 — Asynchronous (Comet & WebSockets)

True Push & Bidirectional

Comet — Server Push

Named by Alex Russell (Dojo Toolkit) in a 2006 blog post. Like a news ticker on TV: the TV station (server) pushes breaking news to your screen without you switching channels or pressing anything.

Definition: Comet enables web servers to send data to the client after only an initial HTTP Request, without the client needing to repeatedly request updates. Minimum dependence between request and response.

  • Uses long-lived HTTP connections — a connection stays open, and the server pushes data when events occur ("server push" / "HTTP streaming")
  • A second connection is needed for the client to upload data to the server
  • Use cases: real-time chat, stock exchange apps, live bidding (eBay), live sports scores
  • Shortcomings: overhead from HTTP headers, inefficiency of long-polling

WebSockets — The Modern Solution

WebSockets are like a phone call instead of sending letters (HTTP). Once connected, both sides can talk freely in real time, simultaneously, without hanging up and calling back for each message.

WebSocket advantages over Comet: No HTTP header overhead per message · True bidirectional (full-duplex) communication · Single connection for both sending and receiving · Part of HTML5

WebSocket Protocol Specifications

IETF RFC 6455

Defines the protocol: how to establish the connection (handshake) and how data is exchanged as frames.

W3C

Defines the JavaScript API: the programming interface developers use in browsers.

WebSocket Handshake — Step by Step

1

Client sends HTTP Upgrade Request

Browser sends a special HTTP GET with headers: Upgrade: websocket, Connection: Upgrade, and a randomly generated Base64-encoded key in Sec-WebSocket-Key.

2

Server responds: 101 Switching Protocols

Server confirms the switch. It concatenates the client's key with the fixed string 258EAFA5-E914-47DA-95CA-C5AB0DC85B11, applies SHA-1 hashing, and Base64-encodes the result as Sec-WebSocket-Accept.

3

WebSocket connection established

The TCP connection is now upgraded to a WebSocket connection. Binary or UTF-8 messages can be exchanged in full-duplex mode on this single connection.

Phase 2 — Smart Summary

High-Yield Summary

🌐 Interaction Layers

  • User ↔ Application: Is the user blocked during updates?
  • Client ↔ Server: Is a new HTTP request needed for every server update?

📌 Model 1: Synchronous (Classic)

  • Paradigm: "Click, Wait, and Refresh"
  • User blocked: Yes
  • HTTP dependence: High — every update = new request + full page response
  • Problems: Slow, loses context, excessive server load, low interactivity

📌 Model 2: Non-Blocking (AJAX)

  • Full name: Asynchronous JavaScript + XML
  • Coined by: Jesse James Garrett, 2005
  • First use: Microsoft Outlook Web Access, 1999
  • Key innovation: AJAX engine as intermediary — decouples user actions from server communication
  • User blocked: No
  • HTTP dependence: Still high — every server update needs a request/response round-trip
  • Heart of AJAX: XMLHttpRequest object
  • Critical readyState: 4 (Complete) + status 200 = success
  • JSON vs XML: JSON preferred — lighter, faster parsing
  • Technologies: XHTML/CSS + DOM + XML/XSLT + XMLHttpRequest + JavaScript

📌 Model 3: Asynchronous (Comet + WebSockets)

  • Comet coined by Alex Russell, March 2006 (Dojo Toolkit)
  • Comet approach: Long-lived connections; server pushes data without client request
  • Comet shortcomings: HTTP header overhead, long-polling inefficiency
  • WebSockets: TCP-based, bidirectional, full-duplex; part of HTML5
  • Protocol spec: IETF RFC 6455
  • API spec: W3C
  • Handshake: HTTP Upgrade → 101 Switching Protocols → WebSocket messages
  • Key response code: 101 Switching Protocols
  • User blocked: No
  • HTTP dependence: Minimum (only initial handshake)

🔑 WebSocket Handshake Details

  • Client sends Upgrade: websocket in HTTP GET request
  • Client generates random Base64-encoded Sec-WebSocket-Key
  • Server concatenates key + 258EAFA5-E914-47DA-95CA-C5AB0DC85B11
  • Server returns SHA-1 hash (Base64-encoded) as Sec-WebSocket-Accept
  • Server responds with HTTP 101 Switching Protocols
  • Connection becomes full-duplex WebSocket over TCP

🪤 Common Exam Traps

⚠️ Trap 1

AJAX is NOT fully asynchronous

AJAX decouples user actions from server communication (user is not blocked), BUT there is still a high degree of dependence between each HTTP Request and Response. Every server-side update still needs a request. True async with minimal dependence = Comet/WebSockets.

⚠️ Trap 2

Default async in open() is FALSE

The signature is open("GET", url, async). The async parameter defaults to false (synchronous). Must explicitly pass true for async behavior.

⚠️ Trap 3

Comet ≠ a specific technology

Comet is a buzzword/umbrella term (like AJAX) for server-push techniques. It doesn't stand for anything and doesn't define a specific technology.

⚠️ Trap 4

WebSocket uses TCP, not HTTP

The handshake uses HTTP, but the subsequent WebSocket connection is a TCP-based protocol — not HTTP anymore. The HTTP connection is "upgraded" to a WebSocket connection.

⚠️ Trap 5

readyState 4 ≠ success alone

You must check BOTH readyState == 4 AND status == 200. readyState 4 just means the request is complete — it could be a 404 or 500 error.

⚠️ Trap 6

Comet needs TWO connections

The long-lived connection is only for server→client data push. A second separate HTTP connection is required for client→server data upload.

  • WebSocket magic string: 258EAFA5-E914-47DA-95CA-C5AB0DC85B11 (concatenated with Sec-WebSocket-Key, SHA-1 hashed, Base64-encoded)
  • AJAX user is NOT blocked, but HTTP Request-Response dependence is still HIGH
  • Comet needs a second connection for client-to-server upload
  • JSON is NOT a technology — it's a lightweight data interchange format
  • async parameter default in open() is false (synchronous!)
  • WebSockets: data exchanged in frames (binary or UTF-8)
  • Comet is also called "server push" or "HTTP streaming"
