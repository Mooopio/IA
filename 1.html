1. The World Wide Web — What Is It?

The World Wide Web (WWW) is a system of hypertext documents viewable via a web browser. It runs on top of the Internet — it's not the Internet itself, but a service that uses Internet protocols.

🔑 Key Definition

Hypertext documents = documents that contain internal cross-references (links) to other documents.

The original Web was built on three pillars:

PillarTechnologyPurpose
Application protocolHTTP (HyperText Transfer Protocol)How browsers and servers communicate
Document formatHTML (HyperText Markup Language)How web pages are structured/described
AddressingURI (Uniform Resource Identifier)How resources are located/addressed
⚠️ Exam Trap

The Web ≠ The Internet. The Internet is the underlying network infrastructure. The Web is one service that runs on it (like email, FTP, etc.).

2. Web Evolution: 1.0 → 2.0 → 3.0

Web 2.0 describes modern web applications accessible via browser (wikis, weblogs, social platforms). Key characteristics:

  • High social component — user-generated content, communities
  • Appealing UIs — enhanced interaction models
  • Protocol improvements: Ajax, Comet/HTTP Streaming, WebSockets
  • New document formats: RSS, Atom, RDF, HTML5
  • New business models — advertising, subscription, freemium
  • Increasing client-side logic — more work done in the browser
📝 Note

Web 2.0 is a technological evolution of Web 1.0.

Web 3.0 (or the Semantic Web) is a vision of semantically interconnected data on the Web. It differs from Web 2.0 in a fundamental way:

Web 2.0Web 3.0
TypeTechnological evolutionConceptual evolution
FocusDocuments + social interactionData (Linked Data)
FormatsRSS, Atom, HTML5RDF, RDFS, OWL, Microformats
🔑 Key Quote (Tim Berners-Lee)

"A 'Semantic Web'...will make it possible for computers to analyze all data on the Web — content, links, and transactions. The day-to-day mechanisms of trade, bureaucracy and our daily lives will be handled by machines talking to machines."

⚠️ Exam Note

Web 3.0 focuses on data not documents — it becomes Linked Data. Key formats: RDF, RDFS, OWL.

3. XML — Extensible Markup Language

XML is a general-purpose meta-language for representation, exchange, and processing of data. It forms the basis of many web-related formats.

📝 Analogy

Think of XML like a universal filing cabinet. You define your own labels (tags) for drawers and folders. Anyone who reads the cabinet knows where everything is because the structure is self-describing.

3.1 XML Tree Structure

XML documents are organized as a tree with elements as nodes:

<?xml version="1.0" encoding="UTF-8"?>
<bookstock>                       ← Root element
  <book id="42">                  ← Child element with attribute
    <title>TCP/IP Illustrated</title>  ← Simple content
    <author>W.R. Stevens</author>
    <isbn>0201633469</isbn>
  </book>
  <book id="43"> ... </book>
<!-- End of document -->          ← Comment
</bookstock>

3.2 XML Terminology (Critical!)

TermDefinitionExample
Root elementThe single top-level element<bookstock>
Parent elementElement containing other elements<book> is parent of title, author, isbn
Child elementElement nested inside another<book> is child of <bookstock>
Sibling elementsElements at the same level (same parent)title, author, isbn are siblings
AttributeName-value pair inside a start tagid="42"
Simple contentText values<title>TCP/IP</title>
Element contentContains further tags<book> contains child tags
Mixed contentBoth text and child elementsCombination of both
Empty contentNo value<br/>

3.3 XML Syntax Rules — "Well-Formed"

⚠️ Exam Favourite

A syntactically correct XML document is called well-formed. Know all the rules:

  • Non-empty elements need both start and end tags: <X>content</X>
  • Empty elements can be self-closed: <X/>
  • Exactly one root element must exist
  • All attribute values must be quoted (single or double)
  • Tags must not overlap: <X><Y></X></Y> is INVALID
  • XML is case sensitive: <X></x> is incorrect
  • Content must comply with the charset (default: UTF-8)

3.4 XML Namespaces

Problem: Two contributors use the same element name <title> to mean different things. Solution: XML namespaces.

🔑 Key Concept

Namespaces are unique identifiers (web addresses) that qualify elements and attributes. A prefix is bound to a namespace and applied to elements: xmlns:b="http://example.com/myNs/" then <b:book>.

<datafile xmlns:a="http://www.example.com/doc/"
          xmlns:b="http://www.example.com/myNs/">
  <a:title>Book list</a:title>     ← doc namespace title
  <b:title>TCP/IP Illustrated</b:title>  ← book namespace title
</datafile>

The namespace URIs don't need to point to real resources — they're just unique identifiers.

3.5 XML Validity

A valid XML document conforms to a type definition. Two approaches:

DTDXML Schema
Full nameDocument Type DefinitionXML Schema
Type supportLimited (PCDATA only)Rich (string, integer, sequences, etc.)
Itself XML?No (different syntax)Yes (XML dialect)
Complex typesNoYes
⚠️ Exam Note

DTD's main shortcoming = limited type definitions. XML Schema fixes this.

4. XHTML

XHTML = Extensible HyperText Markup Language. It is an XML reformulation of HTML.

  • Language constructs deduced from HTML — same expressiveness
  • Stricter syntax → better browser engine standardization
  • Being XML → processable by regular XML tools (e.g., for style or syntax validation)
  • Extensible with other XML languages (e.g., MathML for math expressions)
  • Validated by DTD (e.g., XHTML 1.1 DTD)
  • Empty elements must be self-closed: <br/> (unlike HTML: <br>)

5. DOM vs SAX — XML Processing Models

DOMSAX
Full nameDocument Object ModelSimple API for XML
ApproachTree-based (loads entire document into memory)Event-based (sequential read)
MemoryHigh (whole tree in RAM)Low (processes one element at a time)
NavigationCan traverse tree in any direction, random access by IDSequential only (forward)
Standardized byW3CCommunity standard

📝 DOM Analogy

DOM = reading the entire book and building an index. SAX = someone reads it aloud and you react to each sentence as you hear it.

DOM Example

var elem = document.getElementById("42")
var child = elem.getElementsByTagName("author")[0]
var secondChild = child.childNode
var value = secondChild.nodeValue

SAX Principle

A program registers event handlers at a SAX parser. The parser reads sequentially and fires events (startElement, characters, endElement, etc.) which the handlers react to.

6. Web Interaction Model & Content Types

The Web uses a classic client-server architecture:

  1. Client (browser) requests a document via URI using HTTP
  2. Server sends document with content-type metadata
  3. Client interprets and renders (HTML + CSS + JS)
🔑 Content Type (MIME Type)

Format: Content-Type: type/subtype
Examples: application/xml, video/mpeg, image/jpeg, text/plain

If the browser receives an unknown content type, it looks up an appropriate application in its mapping table and delegates the document to it.

7. JavaScript

JavaScript is an object-based scripting language (standardized as ECMAScript) primarily used for client-side execution.

  • Embedded in or linked from (X)HTML documents
  • Executed by the browser's JavaScript engine
  • Can modify document content or trigger browser actions
  • Often combined with DOM for dynamic document manipulation
⚠️ Critical Security Concept

Sandbox principle: JavaScript running inside the engine can ONLY access objects inside the browser (DOM) by default. It cannot directly access the file system or arbitrary network resources.

8. CSS — Cascading Style Sheets

CSS is a client-side language for defining the layout and visual appearance of markup language elements.

  • Can be embedded directly in (X)HTML or linked externally (<link rel="stylesheet" href="main.css">)
  • One external CSS file can control the layout of all associated pages
  • Structure: selector { property: value; }

CSS Selector Types

SelectorSyntaxExampleTargets
Elementp { ... }p { font-size: 16pt; }All <p> elements
Class.name { ... }.top { padding-top: 20px; }Elements with class="top"
ID#name { ... }#fat { font-weight: bold; }Element with id="fat"

Smart Summary

High-yield bullet points for exam revision

🌐 The Web — Core Pillars

  • HTTP — application protocol (how client and server talk)
  • HTML — document description format
  • URI — document addressing mechanism
  • Web runs on top of Internet protocols (not the Internet itself)

📈 Web Evolution

  • Web 2.0: Social component, Ajax/WebSockets, RSS/Atom, new business models → technological evolution
  • Web 3.0: Semantic Web, data not documents, Linked Data, RDF/RDFS/OWL → conceptual evolution
  • Tim Berners-Lee coined the Semantic Web vision — "machines talking to machines"

📄 XML Key Facts

  • Meta-language for representation, exchange, and processing of data
  • Tree structure: root → parent → child → sibling
  • Content types: empty / simple (text) / element / mixed
  • Well-formed = syntactically correct XML (proper tags, quotes, single root, no overlapping)
  • Valid = well-formed + conforms to DTD or XML Schema
  • DTD: limited types. XML Schema: rich type system (simple + complex types)
  • Namespaces: solve name conflicts using URI prefixes (xmlns:prefix="URI")
  • Default charset: UTF-8

🔄 DOM vs SAX

  • DOM: Tree-based, W3C standard, loads all into memory, allows random access
  • SAX: Event-based, sequential, low memory, uses handler functions (startElement, endElement, characters)
  • DOM → getElementById(), getElementsByTagName()

🏷️ XHTML

  • XML reformulation of HTML → stricter syntax, same expressive power
  • Empty elements: must use <br/> (not <br>)
  • Processable by XML tools, extensible (e.g. MathML)
  • Validated via DTD

🎨 CSS

  • Defines layout and style of markup elements
  • Selectors: element (p), class (.name), ID (#name)
  • External file → change one file to update all linked pages

⚙️ JavaScript

  • Object-based client-side scripting (ECMAScript standard)
  • Executed by browser's JavaScript engine
  • Sandbox principle: only accesses browser objects (DOM) by default — no file system access
  • Combined with DOM → dynamic page manipulation

📬 Content Type (MIME)

  • Format: type/subtype → e.g. application/xml, image/jpeg
  • Sent in HTTP header, tells client how to handle the content
  • If browser can't handle it → delegates to registered application

🗂️ Alternatives to XML

  • JSON: JavaScript Object Notation — lightweight, web APIs (Chapter 3)
  • YAML: YAML Ain't Markup Language — superset of JSON, human-readable, indentation-based hierarchy, cross-references supported
🧠 Memory Triggers

3 Web Pillars: "HAU" — HTTP, Address (URI), doUment format (HTML)
XML validity levels: Well-formed → Valid (DTD or Schema)
DOM vs SAX: DOM = Document in Memory | SAX = Sequential Alert eXecutor
Web generations: 1.0 = Static | 2.0 = Social | 3.0 = Semantic
